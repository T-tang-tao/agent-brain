# Loop 三大陷阱

> Loop 让事情自动化,但也放大了三类风险。Addy Osmani 在原文里专门警告了这些。

## 陷阱 1:理解债(comprehension debt)

### 表现

Loop 跑得越顺,你越不读自己代码。代码库中"你一行都没写过"的代码量飞速增长,你对系统真正理解的深度与代码量之间的差距越来越大。

### 何时发生

- Loop 一周以上不间断运行
- 输出量超过单次 review 能力
- 用户逐渐跳过 review,直接接受产物

### 对策

- **强制 review 节奏**:每 N 次迭代 / 每天 / 每周必读一次产物
- **限制 loop 输出量**:单次 loop 最多生成 X 行代码,超出人工干预
- **关键模块必须读**:核心代码不能跳过 review,即使 loop 觉得"对"

### 检测指标

```text
你的代码库中,你最近 30 天没读过的文件占比:
- < 30%: 健康
- 30-60%: 警告
- > 60%: 危险,需要冻结 loop 重新 review
```

## 陷阱 2:认知投降(cognitive surrender)

### 表现

当 loop 一直运行良好时,你会逐渐失去判断力,照单全收 loop 给的所有结果。从设计师沦为只会按"开始"键的操作员。

### 何时发生

- Loop 长期成功率高,用户开始信任
- 任务超出你的领域专业,你不知道 loop 对不对
- 时间压力下,"让 loop 跑"比"我先想想"更省事

### 对策

- **保持质疑**:loop 输出即使"看起来对",也要问"为什么"
- **保留专业判断**:即使是 loop 跑的代码,关键决策仍由人拍板
- **建立验收标准**:loop 通过的"完成"条件必须有人的明确签字

### 警示信号

- "这个 loop 跑得真好,我都不用看"
- "我也不知道它在做什么,但结果对就行"
- "它说完成了,那应该就是完成了"

## 陷阱 3:Token 黑洞

### 表现

Loop 反复读取上下文、进行多次重试和试探,无论最终是否产出有效结果,Token 都在持续消耗。账单无限制增长。

### 何时发生

- 停止条件无法自动验证,loop 反复尝试
- loop 卡在某个错误上,反复重试同样的事
- 子 agent 数量过多,token 指数级增长

### 对策

#### 1. 强可验证停止条件

```yaml
✅ 好停止条件:
  - "npm test 全绿"
  - "lint 退出码 0"
  - "git diff 为空"
  - "特定 file 包含 string X"

❌ 坏停止条件:
  - "做得更好"
  - "代码看起来合理"
  - "agent 觉得可以了"
```

#### 2. 最大迭代次数

```yaml
loop_config:
  max_iterations: 20  # 超过 20 次 → 自动停下,转人工
  max_token_per_iteration: 50000
  max_total_token: 1000000
```

#### 3. 失败检测

```python
if 连续 3 次相同错误:
    停止 loop,生成错误报告,转人工
```

#### 4. 成本可见

定期看账单:
- 每天
- 每次 loop 结束
- 超过预算告警

## 验证清单

完成 loop 设计/部署后:

- [ ] 是否设置了强验证停止条件(测试/退出码,不是模糊判断)?
- [ ] 是否设置了最大迭代次数?
- [ ] 是否设置了 token 预算上限?
- [ ] 是否设了"连续失败 N 次转人工"机制?
- [ ] 用户是否设定了 review 节奏(每天/每周)?
- [ ] 用户是否知道自己在 loop 中要保留判断,不是照单全收?
- [ ] 是否有"loop 输出量超限 → 暂停"机制?

任何一项 ❌,都有对应陷阱的风险。

## 三个陷阱的优先级

| 陷阱 | 严重性 | 触发难度 | 优先级 |
|------|--------|---------|--------|
| Token 黑洞 | 🔴 高(直接经济损失) | 🟢 低(立刻可见) | P1 必须防 |
| 理解债 | 🟠 中(渐进,难察觉) | 🟡 中(几周后) | P2 长期监控 |
| 认知投降 | 🟠 中(渐进,主观) | 🔴 高(几月后) | P3 团队文化 |

防 P1 靠工程手段(代码 + 配置)。防 P2、P3 靠**团队纪律**——这就是为什么 Osmani 在文末说:

> Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go.
