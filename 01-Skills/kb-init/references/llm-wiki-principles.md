# LLM Wiki 设计原则

> 本文件供 kb-init 技能在 Step 3(设计目录结构)时参考。完整理论见知识库 `00-AgentBase/knowledge/03-LLM-Wiki架构理念.md`。

## 核心思想

LLM 不是被动检索工具,而是**主动知识编译器**——在写入时就把散落的原始材料"编译"成结构化知识,而非提问时临时拼凑。

> **类比**:传统 RAG 像"外卖员"(饿了才取货,现找现拼),LLM Wiki 像"农场主"(在需要之前就把食材种好、加工好、分类存好)。

## 三层架构

设计目录时,基于以下三层思想:

```
Raw Sources(原始层)    →   未加工的原始资料,保留溯源(可选,按需建立)
Wiki(知识层)           →   已编译的结构化知识,人类可读 + Agent 可解析
Schema(架构层)         →   入口(README)、索引、知识地图、AGENTS.md
```

| 层 | 类比 | 说明 |
|---|---|---|
| 原始层 | 食材原产地 | 散落各处,没加工,保留来源 |
| 知识层 | 中央厨房 | 已洗净、切好、按菜谱分类放好 |
| 架构层 | 菜谱标准 | 规定每道菜用什么食材、什么切法 |

## 七条设计原则

1. **先有目录,再填内容**:不要一开始就写很多散文档,先定义目录结构
2. **每个文件只解决一个问题**:不要把所有内容塞进一个大文件
3. **用双链建立知识图谱**:链接的作用不是好看,而是让人和 Agent 都能追踪上下文
4. **写"边界"和"不是什么"**:Agent 容易混淆概念,"不是什么"有时比"是什么"更重要
5. **保留来源,但不要照搬来源**:原始资料 → 摘录 → 清洗 → 归纳 → 写入 wiki → 链接 → 定期复查
6. **区分静态知识和实时状态**:静态知识写进 wiki,实时状态(价格、订单、账户)用工具查
7. **知识和资产分离**:概念解释 vs 可执行模板,分开放
8. **区分人类读者和 Agent 读者**:人类需要的基础概念解释(Agent 天生就懂)单独放 `00-human/`,Agent 直接跳过,节省上下文窗口

## 00-human 模式

知识库中有些内容是**人类入门基础**——概念解释、类比、误解澄清、学习路径。这些对人类是必需的,但 Agent 天生就懂,读了只浪费上下文。

设计原则:
- `00-human/` 放人类入门内容,Agent 不需要读
- 其他模块只保留 Agent 需要的结构化信息(目录映射、项目边界、可执行规则)
- `00-human/` 的写作风格不同:叙述式、讲故事、类比丰富,可以超过 300 行
- 其他模块开头可加重定向提示,指向 `00-human/` 对应的基础概念解释

参考实现:本知识库自身的 `00-AgentBase/00-human/` 目录。

## 知识编译流程

原始材料(对话记录、外部文档、实践笔记)不入库,经"编译"后进入知识库:

```
原始材料              编译过程               知识库文档
(散落、可能矛盾)     ┌──────────────┐      (结构化、消歧)
                   │ 1. 去重归纳     │
对话记录            │ 2. 消歧仲裁     │      wiki/
外部文档    ──────► │ 3. 提炼定义     │ ───► concepts/ ...
实践笔记            │ 4. 建立交叉引用  │
                   │ 5. 补充类比     │
                   └──────────────┘
```

编译原则:
- **矛盾仲裁**:多份材料说法冲突时,以权威来源为准,形成唯一真相
- **去重归并**:同一概念出现在多份材料中,合并为一个页面
- **溯源保留**:编译后的文档标注信息来源,可追溯回原始材料
- **持续累积**:每次编译都是增量更新,知识随时间复利增值

## 防信息孤岛规则

每篇文档发布时必须同时满足:

- **入口≥1**:被索引、知识地图、或至少 1 篇文档引用
- **出口≥1**:文档末尾有相关链接(总览文档除外)
- **零死链**:所有内部链接指向真实存在的文件
- **双向引用**:A 引用 B 时,B 必须回引 A(导航枢纽除外)
