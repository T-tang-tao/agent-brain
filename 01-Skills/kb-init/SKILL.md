# kb-init — 知识库初始化

## When to Use

- 用户说"初始化知识库""建一个知识库""新建 wiki"
- 用户想系统性地管理某领域的知识
- 用户说"帮我整理一个 xxx 知识库 / wiki / 笔记库"
- 用户明确说"初始化知识库"时直接调用

**不适用**:
- 只想写一篇文档,不需要知识库体系
- 已有成熟知识库,只需局部调整
- 只需要向量数据库 / RAG pipeline(那不是 LLM Wiki)

## Procedure

### Step 1: 头脑风暴(不可跳过)

目标:搞清楚用户要什么知识库,不能套模板。

向用户提问,覆盖以下维度。回答模糊时追问,直到能画出目录结构:

1. 这个知识库主要关于什么?一句话描述。
2. 给谁用?只给自己,还是给团队/Agent?
3. 你最常查的问题是什么?(反推目录分类)
4. 知识来源是什么?对话?文档?代码?论文?
5. 有没有实时状态需要管理?(如行情、订单、账户)→ 这些不写进知识库,用工具查
6. 规模预期?几十篇还是几百篇?
7. 有没有已有的散落资料?

**输出**:一段需求摘要(领域类型、读者、来源、规模、特殊约束)。

### Step 2: 确定知识库类型

根据头脑风暴结果归类。参考 `references/kb-types.md` 中的类型表,但不限于这些。如果用户需求跨类型,设计混合结构。

当前支持的知识库类型:

| 类型 | 适用场景 |
|------|----------|
| 编程/项目 | 代码为中心:架构、API、排障、ADR |
| 产品开发 | 产品全生命周期:愿景、需求、设计、发布、复盘 |
| 交易/量化 | 策略、因子、风控(实时数据不入库) |
| 学术/研究 | 论文摘录、概念定义、文献综述 |
| 客服/文档 | FAQ、政策、产品文档(权威来源) |
| 个人助手 | 偏好、习惯、项目笔记(隐私边界) |
| 通用/混合 | 按头脑风暴结果定制 |

### Step 3: 设计目录结构

基于 LLM Wiki 三层思想设计(详见 `references/llm-wiki-principles.md`):

- **原始层**(`raw/`):未加工资料,保留溯源(可选)
- **知识层**:已编译的结构化知识,人类可读 + Agent 可解析
- **架构层**:入口(README)、索引、AGENTS.md
- **资产层**(可选):可执行模板、配置、脚本

设计原则:
- 先有目录,再填内容
- 每个目录职责单一
- 知识和资产分离
- 实时状态不入库
- 目录深度 2-3 层为宜
- **区分人类读者和 Agent 读者**:如果知识库需要大量基础概念解释,考虑建立 `00-human/` 目录,把人类入门内容(Agent 天生就懂的概念)单独隔离,避免浪费 Agent 上下文窗口

**输出**:完整目录树,每个目录附一句话说明。

**需用户确认后**再进入下一步。

### Step 4: 生成 AGENTS.md

为知识库生成专属 AGENTS.md,必须包含 10 个章节:

1. 知识库定位(一句话)
2. 目录结构(完整目录树 + 职责说明)
3. 写作标准(人话优先、类比驱动、可视化、可复述)
4. 文件命名规范
5. 元数据标准
6. 防信息孤岛规则(入口≥1、出口≥1、零死链、双向引用)
7. 知识编译流程
8. 入库工作流
9. 审查清单
10. 版本管理

按知识库类型定制(详见 `references/agents-md-customization.md`):
- 编程库 → 强调 ADR、代码示例规范
- 产品开发库 → 强调需求四要素(背景/目标/验收标准/非目标)、实时进度不入库
- 交易库 → 强调"实时数据不入库"
- 学术库 → 强调引用溯源、概念消歧
- 客服库 → 强调版本控制、权威来源
- 个人助手 → 强调隐私边界、可删除
- **包含 `00-human/` 时** → 补充人类入门区规则:标注 Agent 不读、写作风格不同、300 行限制豁免(详见 `references/agents-md-customization.md` 的"00-human 人类入门区规则")

**需用户确认后**再进入下一步。

### Step 5: 生成入口文件

| 文件 | 用途 |
|------|------|
| `README.md` | 入口:定位 + 目录结构 + 快速入口 + 推荐阅读顺序 |
| `00-目录索引.md` | 导航:按目录列出所有文档,附一句话说明 |
| `AGENTS.md` | 管理员规范(Step 4 生成) |

### Step 5.5: 应用 `.agent/` 项目级规范(v1.1.0+)

**适用范围**:用户说"用这个知识库作为一个项目""集成到我的项目里"时。

**默认**:kb-init 生成的"知识库项目"也是一个项目,应该按 Agent 项目的 `.agent/` 规范创建项目级目录:

```
<项目根>/
├── AGENTS.md              ← 项目指令(Step 4 生成,放项目根)
├── README.md              ← 项目入口
├── 00-目录索引.md
├── .agent/                ← 项目级资产
│   ├── skills/            ← 项目特定技能(平铺)
│   ├── agents/            ← 项目特定子代理
│   ├── mcp/               ← 项目 MCP 配置
│   ├── memory/            ← 项目长期记忆
│   └── hooks/
└── <知识库目录>/
```

**核心**:
- **AGENTS.md 在项目根**(不放在知识库子目录)
- **`.agent/` 在项目根**(统一目录名,跨 Runtime 一致)
- **知识库目录**(00-XX/)是知识层的物理体现,与 `.agent/` 平级

**完整规范**:[00-项目级配置](../00-AgentBase/runtime/00-项目级配置.md)

**为什么**:换 Runtime(Claude Code / Codex / Hermes)不需要改目录结构;团队成员各用各的 Runtime 但项目结构一致;部署脚本可复用。

### Step 6: 防信息孤岛检查

验证知识库骨架:
- 每个目录都有 README 或被索引收录(入口≥1)
- README 和索引中有指向各目录的链接(出口≥1)
- 所有内部链接指向真实存在的文件(零死链)
- 从任一文件出发,能找到回首页和索引的路径(导航闭环)

### Step 7: 交付

向用户展示:
1. 需求摘要(头脑风暴结论)
2. 目录结构树
3. 生成的文件清单
4. 后续填充建议(先填哪些、怎么编译、怎么积累)

## Pitfalls

- **跳过头脑风暴直接套模板**:不同类型知识库目录结构不同,必须先理解需求
- **把实时状态写进知识库**:价格、订单、账户余额等用工具查,不写死
- **AGENTS.md 照搬通用模板**:必须按知识库类型定制重点章节
- **初始化时预填大量内容**:只做骨架,不预填内容文件
- **目录太深**:超过 3 层会让导航困难,人和 Agent 都难定位
- **忘记防孤岛检查**:初始化时就埋下孤岛,后续越来越难修
- **人类基础知识和 Agent 检索内容混放**:Agent 天生就懂的概念(如"什么是 Agent")不需要写进 Agent 要读的模块,应放 `00-human/` 或省略

## Verification

- [ ] 头脑风暴完成?能用一句话说清用户要什么知识库?
- [ ] 目录结构根据需求定制,非套用模板?
- [ ] 知识和资产分离?
- [ ] 实时状态被排除在知识库之外?
- [ ] 人类基础知识和 Agent 检索内容已分离?(如有 `00-human/`,AGENTS.md 中已补充规则?)
- [ ] AGENTS.md 包含全部 10 个章节,且按类型定制?
- [ ] 入口文件(README、索引)已生成?
- [ ] 防信息孤岛检查通过?(入口≥1、出口≥1、零死链、导航闭环)
- [ ] **如作为项目集成,已应用 `.agent/` 规范(AGENTS.md 在项目根 + `.agent/` 目录创建)?**
- [ ] 向用户输出了需求摘要和后续建议?
