# Agent 边界与限制

> 基础概念解释见 [00-human/03-常见误解与澄清](../00-human/03-常见误解与澄清.md)。
>
> 本文件聚焦项目特定的边界规则和风险分级。

## 一句话解释

Agent 越能行动，越需要边界。边界用来规定 Agent 什么可以做、什么需要确认、什么禁止做、完成后如何验证。

## 为什么 Agent 需要边界

普通模型主要输出文本，风险通常来自“说错”。

Agent 可以调用工具，风险会扩大到“做错”：

- 改错文件；
- 删除数据；
- 执行危险命令；
- 调用外部 API；
- 写入错误记忆；
- 泄露敏感信息；
- 在未确认时提交、推送、下单或发送消息。

所以 Agent 不能只靠“聪明”，还需要权限和边界。

## Agent 可能做错什么

常见风险：

| 风险 | 例子 |
|---|---|
| 理解错误 | 用户只是询问，Agent 却直接修改文件 |
| 工具误用 | 不该运行删除命令却运行了 |
| 范围扩大 | 用户只要求改一个文件，Agent 顺手重构很多文件 |
| 上下文过期 | 根据旧文档或旧代码做判断 |
| 未验证完成 | 没跑测试就声称完成 |
| 记忆污染 | 把一次性信息写成长期记忆 |
| 外部副作用 | 调用发送消息、交易、部署、支付等不可轻易撤销操作 |

## 工具权限边界

工具权限是 Agent 边界的第一层。

常见策略：

- 只读工具默认允许；
- 文件写入需要在任务范围内；
- shell 命令按风险分级；
- 外部 API 需要确认；
- 删除、推送、部署、交易等高风险动作必须确认。

## 文件编辑边界

文件编辑常见规则：

- 只修改任务相关文件；
- 修改前尽量先读文件；
- 不随意重构无关代码；
- 不覆盖未知内容；
- 不修改密钥、凭证、生产配置，除非用户明确要求；
- 修改后说明改了哪些文件。

## Shell 命令边界

Shell 命令风险较高，因为它能影响系统状态。

通常可以允许：

- 查看文件；
- 运行测试；
- 安装已确认依赖；
- 运行项目脚本。

通常需要确认：

- 删除文件；
- 重置 git 状态；
- 推送代码；
- 部署；
- 修改系统配置；
- 执行长时间或高成本命令。

## 外部 API 边界

外部 API 可能产生真实世界影响。

需要特别谨慎：

- 发送邮件；
- 发送 Slack/IM 消息；
- 创建 issue 或 PR；
- 修改数据库；
- 触发部署；
- 下单、交易、支付；
- 调用生产服务。

原则：

```text
只读查询可以更宽松；写入和外部副作用需要确认。
```

## 记忆写入边界

Memory 很容易被污染。

不应该写入：

- 临时对话；
- 一次性任务细节；
- 未确认结论；
- 密钥、token、密码；
- 可以从代码或 git 历史直接得出的事实。

适合写入：

- 用户长期偏好；
- 项目非显然约束；
- 已确认的工作方式；
- 反复出现的踩坑经验；
- 对未来任务有价值的规则。

## 什么时候必须问人

通常必须问人：

- 操作不可逆；
- 会影响外部系统；
- 会花钱；
- 会删除或覆盖数据；
- 会改变项目范围；
- 用户意图不明确；
- 多个合理方案且取舍依赖用户偏好。

## 什么时候可以自主执行

通常可以自主执行：

- 读取文件；
- 搜索代码；
- 运行只读命令；
- 修改明确任务范围内的文件；
- 运行测试；
- 修复测试暴露的直接问题；
- 做可逆的小调整，并在总结中说明。

## 验证的重要性

Agent 完成任务后不能只说“应该可以”。

应尽量验证：

- 测试是否通过；
- lint/typecheck 是否通过；
- 输出文件是否存在；
- 引用来源是否真实；
- 回测是否无未来函数；
- 外部状态是否符合预期。

如果无法验证，应该如实说明：

```text
已完成修改，但没有运行测试，因为当前环境缺少依赖。
```
