# 权限分级模型

> 本文给出 Agent 工具权限的 5 级分类,作为 [`01-Agent边界与限制.md`](./01-Agent边界与限制.md) 的具体落地模型。
>
> 适用于:Claude Code / Codex / Hermes 的权限配置。

## 1. 5 级权限模型

```
L0 ─ 只读(无确认)
    ├─ Read/Glob/Grep/View
    ├─ 网络搜索(Web Search)
    └─ 调用文档/参考知识(检索)
    
L1 ─ 本地读+轻写(无确认)
    ├─ 编辑已存在的本地文件(Edit/Update)
    ├─ 在工作目录创建新文件
    └─ 运行只读命令(ls/cat/grep/find)
    
L2 ─ 受控写(需白名单)
    ├─ 修改 .agent/ 内文件
    ├─ 修改项目配置文件
    ├─ 安装依赖(npm install / pip install)
    └─ 启动本地服务(uvicorn/python -m http.server)
    
L3 ─ 系统级写(需白名单+默认确认)
    ├─ 写 .git/ 或 git commit/push
    ├─ 执行 Bash 命令(Bash 工具)
    ├─ 修改环境变量 / 系统设置
    └─ 跨项目写操作
    
L4 ─ 破坏性 / 不可逆(永远确认)
    ├─ 删除文件或目录(rm)
    ├─ 删数据库表(drop table)
    ├─ 强制推送(force push)
    ├─ 执行未知脚本
    └─ 真实 API 写操作(转账、下单、修改外部状态)
```

## 2. 各级处理策略

| 级别 | 是否需用户确认 | 默认行为 | 例外方式 |
|------|---------------|----------|----------|
| L0 只读 | 否 | 直接执行 | - |
| L1 本地轻写 | 否(可关闭) | 直接执行 | `permissions.deny` 禁用 |
| L2 受控写 | 默认需白名单 | 询问 | `permissions.allow` 白名单 |
| L3 系统级 | 每次都询问 | 询问 | 永久白名单(慎用) |
| L4 破坏性 | 永远询问 + 二次确认 | 询问 | 几乎不应给永久白名单 |

## 3. 各 Runtime 配置示例

### 3.1 Claude Code(~/.claude/settings.json)

```json
{
  "permissions": {
    "allow": [
      "Read(./.agent/**)",
      "Edit(./src/**)",
      "Bash(npm test:*)",
      "Bash(git status)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(curl * -X POST:*)",
      "Write(/etc/**)",
      "Edit(/Users/**/.ssh/**)"
    ]
  }
}
```

### 3.2 Codex CLI(~/.codex/config.toml)

```toml
[permissions]
allow = [
    "read(./.agent/**)",
    "edit(./src/**)",
    "shell(npm test)",
    "shell(git status)"
]
deny = [
    "shell(rm -rf *)",
    "shell(curl * -X POST *)"
]
```

## 4. 项目级 .agent/ 权限设置

按本知识库的 `.agent/` 规范,在项目级配置:

```
.agent/
├── permissions.json      ← 项目级权限规则
└── boundaries/           ← 项目级边界策略
```

`.agent/permissions.json`(参考):

```json
{
  "allow": [
    "Read(./.agent/**)",
    "Read(./src/**)",
    "Edit(./src/**)",
    "Edit(./.agent/skills/**)",
    "Bash(npm test:*)",
    "Bash(pytest:*)"
  ],
  "deny": [
    "Bash(rm -rf:*)",
    "Edit(./.env)",
    "Edit(./secrets/**)",
    "Write(/etc/**)",
    "Write(./package-lock.json)"
  ],
  "ask": [
    "Bash(git push:*)",
    "Edit(./.github/**)",
    "Bash(curl * -X POST:*)"
  ]
}
```

## 5. 决策矩阵

| 工具类别 | 默认级别 | 推荐提升到 |
|----------|---------|------------|
| Read/Glob/Grep | L0 | L0(保持) |
| Web Search/Fetch | L0 | L1(若会触发 API 计费) |
| Edit 已存在文件 | L2 | L1(项目内文件) |
| Write 新文件 | L2 | L1(在 .agent/skills 内) |
| Bash 简单命令 | L3 | L2(白名单具体命令) |
| Bash 任意命令 | L3 | 保持(询问) |
| git commit | L3 | L2(白名单) |
| git push | L3 | L3(询问) |
| 删除文件 | L4 | L4(永远询问) |
| 外部 API 写操作 | L4 | L4(永远询问+人工二次确认) |

## 6. 实际操作建议

### 6.1 起步阶段(信任)

- L0-L1 全开,无需确认
- L2 大部分白名单(只 ban rm -rf 等明显危险)

### 6.2 熟悉阶段(谨慎)

- 每次 L3 操作都询问
- 收集"该询问但不该永久拒绝"的工具,加入白名单

### 6.3 长期阶段(自动)

- 已有白名单覆盖大部分常规工作
- L4 操作永远询问
- 定期审计白名单

## 7. 常见反模式

| 反模式 | 后果 | 替代 |
|--------|------|------|
| L4 操作给永久白名单 | 一次失误,不可挽回 | 永远 L4 |
| Bash 全开 | 注入风险高 | 限制具体命令 |
| 不记录权限审计 | 不知道谁什么时候做了什么 | 开启 trace + 审计日志 |
| 跨项目共享配置 | 权限泄露 | 项目级 .agent/permissions.json |
