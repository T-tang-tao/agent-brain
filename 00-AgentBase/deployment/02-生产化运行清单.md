# 生产化运行清单

## 一句话解释

生产化 Agent 不是“能跑起来”就够了,而是要能被配置、监控、审计、回滚和持续评估。

## 生产化目标

| 目标 | 含义 |
|---|---|
| 可控 | 权限、边界、确认策略明确 |
| 可观测 | 能看到输入、工具调用、结果和错误 |
| 可回滚 | 关键写操作有撤销或补偿方案 |
| 可评估 | 有固定任务集衡量质量变化 |
| 可维护 | Prompt、Skill、工具、模型版本可管理 |
| 可迁移 | Runtime 或模型切换时有清单 |

## 上线前清单

- [ ] 明确 Agent 目标和不负责的范围。
- [ ] 明确允许的工具和禁止的工具。
- [ ] 明确哪些操作需要人确认。
- [ ] 配置密钥隔离,不把 token 写入 prompt 或仓库。
- [ ] 配置日志脱敏。
- [ ] 准备最小评估任务集。
- [ ] 准备失败和回滚流程。
- [ ] 准备版本记录:模型、Runtime、Prompt、Skill、MCP。

## 运行时配置

| 配置 | 要求 |
|---|---|
| 模型 | 记录名称、版本、用途、降级模型 |
| Prompt | 版本化,记录变更原因 |
| Skill | 来源明确,状态标记 |
| Tool / MCP | 权限分级,错误格式统一 |
| Memory | 写入策略明确,支持清理 |
| 日志 | 可追踪工具调用和决策摘要 |
| 审批 | 高风险动作强确认 |

## 监控指标

| 指标 | 说明 |
|---|---|
| 成功率 | 任务是否完成 |
| 一次通过率 | 是否无需人工修正 |
| 工具失败率 | 工具参数错误、超时、权限失败 |
| 人工接管率 | 需要人介入的比例 |
| 成本 | token、工具、外部 API 成本 |
| 延迟 | 单轮响应和端到端耗时 |
| 回滚次数 | 写操作造成问题的次数 |
| 安全拦截次数 | 边界规则触发频率 |

## 事故分级

| 等级 | 示例 | 处理 |
|---|---|---|
| P0 | 删除生产数据、泄露密钥、越权外发 | 立即停用、保全日志、人工恢复 |
| P1 | 错误修改关键文件、错误调用外部系统 | 暂停写权限、回滚、复盘 |
| P2 | 任务失败、工具频繁报错 | 降级模型或工具,修复配置 |
| P3 | 输出格式不稳定、体验差 | 调整 prompt 或评估集 |

## 版本管理

每次重要变更记录:

- 日期。
- 变更对象:模型 / Prompt / Skill / Tool / Runtime / 权限。
- 变更原因。
- 影响范围。
- 验证结果。
- 回滚方式。

## 发布策略

推荐灰度:

1. 本地只读验证。
2. 测试环境可写验证。
3. 小范围真实任务。
4. 扩大到团队。
5. 纳入持续监控。

不要第一次上线就给生产写权限。

## 退场机制

Agent 必须知道什么时候停下来:

- 目标不清。
- 权限不足。
- 工具连续失败。
- 检测到高风险操作。
- 发现用户假设与事实冲突。
- 验证失败且无法自行定位。

停下来不是失败,是生产系统的安全能力。
