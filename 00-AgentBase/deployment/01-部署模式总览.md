# Agent 部署模式总览

> Agent 部署和普通应用部署不同——它涉及 Runtime、模型、工具链、记忆系统、可观测性等多组件协调。本文档讲清楚 Agent 部署的核心概念、模式选择和常见架构。

## 一句话理解

Agent 部署就是"把 Agent 跑起来"——选择 Runtime、配置环境、接入模型、挂载工具、设置可观测性,确保它能稳定运行。

## 为什么 Agent 部署是独立话题

传统应用部署:代码 + 运行时 + 依赖 → 跑起来就行。

Agent 部署复杂得多,因为它涉及:

| 组件 | 部署时要决定什么 |
|------|----------------|
| **Runtime** | 选 Hermes / Claude Code / Codex / 其他,装哪个版本 |
| **Model** | 用哪个 LLM(provider、版本、是否私有部署) |
| **Tools** | 接哪些 MCP server、文件权限范围、网络访问范围 |
| **Memory** | 长期记忆存哪里、谁能读写、是否跨实例共享 |
| **Observability** | 日志输出到哪、是否接 OTel、告警通道 |
| **Sandbox** | 沙箱规则、文件隔离、网络隔离 |
| **Approval** | 哪些操作要审批、谁有审批权限 |

任何一个没配好,Agent 就跑不稳或跑不安全。

## 三种典型部署模式

### 模式 1:本地交互式(单机个人使用)

```
┌────────────────────────────────┐
│ 开发者本地                       │
│ ┌──────────────────────────┐   │
│ │ Claude Code / Codex CLI   │   │
│ │   ↓                       │   │
│ │ LLM API (云端)            │   │
│ │   ↑                       │   │
│ │ 本地文件系统               │   │
│ │ 本地 MCP server (可选)     │   │
│ └──────────────────────────┘   │
└────────────────────────────────┘
```

**适用场景**:个人开发、临时任务、学习探索
**特点**:最简单,无服务器成本,但只能单人用

### 模式 2:服务端长驻(共享服务)

```
┌─────────────────────────────────────────────┐
│              服务器                          │
│                                              │
│  ┌──────────┐    ┌──────────────────────┐  │
│  │ Hermes   │ ←→ │ LLM API              │  │
│  │ Gateway  │    └──────────────────────┘  │
│  │   ↓      │    ┌──────────────────────┐  │
│  │ Agent    │ ←→ │ 共享记忆 (SQLite/PG)  │  │
│  │ Workers  │    └──────────────────────┘  │
│  │   ↓      │    ┌──────────────────────┐  │
│  │ 工具调度  │ ←→ │ MCP servers          │  │
│  └──────────┘    └──────────────────────┘  │
│                                              │
│  外部接入: Slack/飞书/邮件/Webhook            │
└─────────────────────────────────────────────┘
```

**适用场景**:团队共享 Agent 服务、定时任务、消息网关
**特点**:多用户、可持久化、可监控,但运维复杂

### 模式 3:嵌入式集成(嵌入业务系统)

```
┌────────────────────────────────────────────┐
│            业务应用                          │
│  ┌──────────────────────────────────────┐ │
│  │ 主应用                                 │ │
│  │   ↓ 内部调用                          │ │
│  │ ┌────────┐    ┌──────────────────┐   │ │
│  │ │ Agent  │ ←→ │ LLM API          │   │ │
│  │ │ 运行时  │    └──────────────────┘   │ │
│  │ │ (嵌入) │    ┌──────────────────┐   │ │
│  │ │        │ ←→ │ 业务工具 (内置)   │   │ │
│  │ └────────┘    └──────────────────┘   │ │
│  └──────────────────────────────────────┘ │
└────────────────────────────────────────────┘
```

**适用场景**:产品内嵌 AI 能力、自动化工作流
**特点**:深度集成,但耦合度高,Agent 升级会影响主应用

## 部署环境管理

### 三套环境原则

| 环境 | 用途 | 模型选择 | 数据隔离 |
|------|------|---------|----------|
| **开发** | 本地开发、调试 | 便宜模型(GPT-3.5/小模型) | 沙箱 + 测试数据 |
| **预发** | 上线前验证 | 生产同型号 | 脱敏生产数据 |
| **生产** | 真实服务 | 生产型号 | 真实数据 + 严格审计 |

### 环境隔离机制

```
┌─────────────────────────────────────────────┐
│              环境隔离                          │
│                                              │
│  ┌────────────┐                              │
│  │ 文件系统    │ 沙箱 + 目录隔离               │
│  │ 网络        │ firewall + 出口白名单         │
│  │ 进程        │ 容器/namespace 隔离           │
│  │ 密钥        │ 环境变量 + secrets manager     │
│  │ 记忆        │ 独立存储实例                   │
│  │ 工具        │ 不同 MCP server 配置           │
│  └────────────┘                              │
└─────────────────────────────────────────────┘
```

## 各 Runtime 的部署模式对比

| Runtime | 本地交互式 | 服务端长驻 | 嵌入式集成 | 推荐场景 |
|---------|----------|-----------|-----------|----------|
| Hermes Agent | ✓ | ✓✓✓(原生) | ✓ | 团队服务、消息网关 |
| Claude Code | ✓✓✓ | ✗ | △ | 个人开发、IDE 集成 |
| Codex CLI | ✓✓✓ | ✗ | △ | 个人开发、CI 脚本 |

> **关键**:Claude Code 和 Codex CLI 设计上就是交互式 CLI,不适合长驻服务。需要长驻服务时,选 Hermes Agent 或自建。

### Hermes Agent 的服务端部署

Hermes 原生支持服务端长驻模式,通过 [Hermes 消息网关](../runtime/02-Hermes-Agent/08-消息网关.md) 接入外部系统。详见 [Hermes 安装部署](../runtime/02-Hermes-Agent/02-安装部署.md)。

### Claude Code 的部署

Claude Code 主要用于本地交互式,通过 [Claude Code IDE 集成](../runtime/03-ClaudeCode/11-IDE与平台集成.md) 嵌入 IDE。详见 [Claude Code 安装部署](../runtime/03-ClaudeCode/02-安装部署.md)。

### Codex CLI 的部署

Codex CLI 主要用于本地交互式,通过 [Codex MCP 集成](../runtime/04-Codex-CLI/05-MCP集成.md) 接入外部工具。详见 [Codex 安装部署](../runtime/04-Codex-CLI/02-安装部署.md)。

## CI/CD 中的 Agent

Agent 也常用于 CI/CD 流水线:

| 场景 | 怎么做 |
|------|--------|
| 代码评审 | PR 触发 Agent 自动 review |
| 测试生成 | Agent 读代码生成单元测试 |
| 文档同步 | Agent 读代码更新文档 |
| Issue 处理 | Agent 自动 triage 简单 issue |
| 发布辅助 | Agent 生成 changelog、release notes |

部署 Agent 到 CI 时要注意:
- 用非交互模式(如 Codex 的 `--non-interactive`)
- 配置只读沙箱
- 限制网络访问
- 输出结构化结果供流水线消费

## 部署的安全考虑

| 风险 | 缓解 |
|------|------|
| 密钥泄露 | 用 secrets manager,绝不写入镜像或配置文件 |
| 越权操作 | 沙箱 + 审批 + 最小权限 |
| 数据外泄 | 网络出口白名单 + 文件系统隔离 |
| 资源耗尽 | rate limit + cost limit + timeout |
| 提示注入 | 输入消毒 + 工具权限限制 |

详见 [Agent 边界与限制](../safety/01-Agent边界与限制.md)。

## 部署的可观测性

部署后必须配置:

| 必须 | 推荐 | 可选 |
|------|------|------|
| 日志输出 | 指标采集 | 用户行为分析 |
| 错误告警 | 成本告警 | 性能追踪 |
| 健康检查 | 延迟告警 | A/B 测试 |

详见 [可观测性](../behavior/06-可观测性(Observability).md)。

## 部署的版本管理

| 项 | 建议 |
|----|------|
| Runtime 版本 | 用稳定渠道,记录在配置仓库 |
| 模型版本 | 锁定具体版本,升级走基准测试 |
| Prompt 版本 | 入库管理,每次修改有记录 |
| Skill 版本 | 入库管理,与 Runtime 版本对应 |
| 配置版本 | 入库管理,环境间差异明文化 |

详见 [迁移 Playbook](../06-Migration/README.md) 的版本升级章节。

## 部署检查清单

部署前过一遍:

- [ ] Runtime 装好,版本对齐
- [ ] LLM API key 配置好(放 secrets,不进 git)
- [ ] 沙箱规则配置好(文件、网络、进程)
- [ ] 工具权限配置好(只给必要的 MCP server)
- [ ] 记忆系统配置好(独立实例)
- [ ] 日志输出配置好(结构化,输出到聚合系统)
- [ ] 告警配置好(成本、延迟、错误)
- [ ] 健康检查配置好(Runtime 启动后能自动检测)
- [ ] 回滚方案准备好(知道怎么快速回退到上一版本)
