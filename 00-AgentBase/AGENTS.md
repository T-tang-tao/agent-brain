# AGENTS.md — Agent 知识库管理员规范

> 本文件是 `00-AgentBase/` 知识库的**管理员规范**。
>
> 管两件事:**入库标准**(Schema:目录结构、字段规范、命名规则)和**管理流程**(编译、入库、审查、沉淀、版本管理)。
>
> 部署、编排与管理指南见 [根目录 AGENTS.md](../AGENTS.md)。

## 0. 知识库定位

`00-AgentBase/` 是 Agent 相关的**完整知识库**,作为整个系统的"大脑":

| 视角 | 用途 |
|------|------|
| 人类 | 理解 Agent 是什么、怎么搭、怎么编排 |
| Agent | 作为 wiki 被检索,提供认知基础 |
| 部署 | 根目录 AGENTS.md 基于本知识库搭建、部署、编排 Agent 环境 |

### 三层架构映射

对应 Karpathy LLM Wiki 三层理论:

```
Karpathy 三层              本知识库
────────────              ────────────
Raw Sources(原始资料)  →   对话/外部文档/实践笔记(不入库,经编译后进入)
Wiki(知识层)           →   00-AgentBase/(认知层=高阶页面)
Schema(约束层)         →   本文件(管理员规范=入库标准+管理流程)
```

核心思想:LLM 不是被动检索工具,而是**主动知识编译器**——在写入时就把散落的原始材料"编译"成结构化知识,而非提问时临时拼凑。

## 1. 目录结构

根级放总览(不归属单一模块的全局性内容),七个子目录**直接对应知识地图的七大模块**,各目录内部独立编号:

```
00-AgentBase/
│
│  ── 根级:总览 ──
├── 00-目录索引.md           # 导航
├── 01-AgentBase总览.md       # 结构 + 知识库入口
├── 02-概念全景辨析.md         # 跨模块概念边界(项目特定)
├── 03-知识地图.md             # 全景导航
│
│  ── 人类入门(Agent 不需要读) ──
├── 00-human/        # 人类基础认知:Agent 是什么、术语速览、常见误解、使用指南
│
│  ── 七大模块(对应知识地图) ──
├── model/          # Model:理解、推理、生成、决策
├── runtime/        # Runtime:CLI、Harness、Context、Permission
├── tools/          # Tools:Built-in、MCP、Custom
├── knowledge/      # Knowledge:Wiki、RAG、Memory、Project Docs
├── behavior/       # Behavior:Prompt、Skill、Policy
└── safety/         # Safety:Permission、Boundary、Verification
```

**分类原则**:知识地图 = 目录结构。跨模块的全局性内容(定义、辨析、导航)放根级,单一模块的深入认知放对应子目录。各子目录内部从 01 起独立编号,互不影响。

**00-human 特殊规则**:
- 放人类需要但 Agent 不需要读的**基础认知**(概念解释、类比、误解澄清、学习路径)
- Agent 天生就懂这些概念,读这些内容只浪费上下文窗口
- 00-human 的文档风格与其他模块不同:叙述式、讲故事、类比丰富,可以超过 300 行
- 其他模块的文档只保留 Agent 需要的结构化信息(目录映射、项目边界、可执行规则)

## 3. 写作标准

### 3.1 六条核心原则

- **人话优先**:不堆术语,首次出现必须用日常语言解释
- **类比驱动**:每个概念配一个现实类比(如"Skill = 技能包","MCP = 标准化插座")
- **可视化**:关键关系用文本图 / 表格呈现,不要纯文字墙
- **可复述**:读完任一文档,能用 3 句话向同事讲清楚核心
- **不写操作步骤**:认知层**不放**具体命令、配置、代码(那些属于实践层)
- **交叉引用**:提及实践层概念时,链接到对应目录,但不展开
- **链接格式**:跨文档引用一律用标准 markdown `[显示文字](相对路径.md)`,**禁止**:
  - Obsidian wikilink `[[path]]`(双层括号、纯 Obsidian 生态、其它渲染器不识别)
  - 裸 `[path]`(单层括号无 URL、渲染为普通文本不可点击)
  - 写死绝对路径 `D:\valut\agent\...`(换机器即失效)
  - 显示文字和 URL 重复(浪费字符、阅读时冗余)— 文字取文件名,URL 用相对路径

### 3.2 文档结构建议

```markdown
# [概念名称]

> 一句话:这个概念是什么、解决什么问题。

## 这是什么
(日常语言解释,配类比)

## 为什么需要它
(没有它会怎样)

## 它和 X 有什么区别
(对比表,划清边界)

## 全景中的位置
(在知识地图里属于哪个模块,和谁关联)

## 深入了解
(可选:更细的子概念、演进脉络)
```

### 3.3 语态要求

| 场景 | 用 | 不用 |
|------|-----|------|
| 解释概念 | "Agent 是一个能…" | "执行以下步骤来创建 Agent…" |
| 对比辨析 | "X 和 Y 的区别在于…" | "你应该先配置 X 再配置 Y…" |
| 提及实践层 | "具体配置见 `01-Skills/`" | "运行 `npm install xxx` 安装…" |

## 4. 禁止事项

- ❌ 放可直接执行的代码块(示例代码除外,需标注"仅示意")
- ❌ 放真实配置文件、密钥、环境变量
- ❌ 写成 API 文档或操作手册
- ❌ 超过 300 行(人类注意力有限,`00-human/` 除外——人类入门内容可以更长)
- ❌ 未经解释的缩写和术语堆砌
- ❌ 把实践层的 SOP / 配置 / 脚本搬进认知层

## 5. 文件命名规范

### 5.1 认知层特例

认知层文档(00)**允许使用中文文件名**以提升可读性(如已有的 `01-AgentBase总览.md`)。

### 5.2 编号规则

- **根级**:00-03 连续编号,表示总览类(定义、辨析、导航),不归属单一模块
- **00-human**:01-04 连续编号,表示人类入门类(基础认知、术语、误解、使用指南)
- **子目录**:各目录内部从 01 起独立编号,互不影响
- 新增文档时,在对应目录末尾续号

### 5.3 特殊文件

| 文件名 | 用途 |
|--------|------|
| `01-AgentBase总览.md` | 根级入口,定义 + 目录结构 + 快速入口(替代传统 README) |
| `AGENTS.md` | 本文件,知识库 Schema |
| `00-目录索引.md` | 知识库导航页 |

## 7. 知识编译与入库流程

### 7.1 编译:从原始材料到知识库

原始材料(对话记录、外部文档、实践笔记)不入库,经"编译"后进入知识库:

```
原始材料              编译过程               知识库文档
(散落、可能矛盾)     ┌──────────────┐      (结构化、消歧)
                   │ 1. 去重归纳     │      00-AgentBase/
对话记录            │ 2. 消歧仲裁     │      model/ runtime/
外部文档    ──────► │ 3. 提炼定义     │ ───► knowledge/ ...
实践笔记            │ 4. 建立交叉引用  │
                   │ 5. 补充类比     │
                   └──────────────┘
```

编译原则:

| 原则 | 操作 |
|------|------|
| 矛盾仲裁 | 多份材料冲突时,以权威来源为准 |
| 去重归并 | 同一概念出现在多份材料中,合并为一个页面 |
| 溯源保留 | 文档标注信息来源,可追溯回原始材料 |
| 持续累积 | 每次编译都是增量更新,知识随时间复利 |

### 7.2 沉淀:从知识库到实践层

当知识库中的概念被反复引用但缺乏实操指南时,触发沉淀到实践层(`01-06/`):

```
知识库文档               沉淀过程                实践层文档
"Agent 可以通过         ┌──────────┐         01-Skills/
 Skill 扩展能力"  ──────► │ 1. 提炼模式 │ ──────►  skill-definition.md
                        │ 2. 写成模板 │
                        │ 3. 实际验证 │
                        │ 4. 补充排障 │
                        │ 5. 建立关联 │
                        └──────────┘
```

沉淀触发条件:

| 触发信号 | 动作 |
|----------|------|
| 同一个问题被问 3 次以上 | 沉淀为实践层 SOP |
| 知识库文档里的概念被反复引用但无实操指南 | 补充实践层文档 |
| 实践中反复试错才跑通的流程 | 沉淀为 checklist |
| 临时方案验证有效 | 沉淀为正式模板 |

### 7.3 质量门槛

从原始材料到实践层发布,必须跨越以下门槛:

```
理解(能说清) ──► 验证(跑通过) ──► 规范(可复现) ──► 就绪(他人可用)
   知识库            个人环境          团队环境           实践层发布
```

**禁止跳级**:不能从"理解"直接跳到"实践层发布",必须经过"验证"和"规范"两步。

### 7.4 入库工作流

```
发现知识缺口 / 获取新原始材料
    ↓
确认属于知识库范畴(是概念解释,不是操作指南)
    ↓
确定归属:根级 or 七大模块之一
    ↓
起草(叙述式,遵循第 3 节写作标准)
    ↓
自检(对照第 9 节审查清单 + 防孤岛检查)
    ↓
补充连接:更新索引收录(入口) + 文档末尾加相关链接(出口)
    ↓
审阅(以"外行能否看懂"为标准)
    ↓
发布 → 更新 00-目录索引.md → 检查双向引用
```

## 8. 与实践层的关联

知识库概念和实践层资产是**一对多**的关系:一个概念可能对应多个实践模板。

| 知识库模块 | 实践层目录 | 关联方式 |
|------------|-----------|----------|
| behavior → Skill 概念 | [`01-Skills/`](../01-Skills/) | 知识库解释"Skill 是什么",实践层提供"怎么定义一个 Skill" |
| tools → Plugin 概念 | [`02-Plugins/`](../02-Plugins/) | 知识库解释"Plugin 是什么",实践层提供"怎么装、怎么配" |
| behavior → Prompt 概念 | [`03-Prompts/`](../03-Prompts/) | 知识库解释"Prompt 的作用",实践层提供"经过验证的模板" |
| tools → MCP 概念 | [`04-MCP/`](../04-MCP/) | 知识库解释"MCP 协议",实践层提供"server 配置示例" |
| safety → Boundary 概念 | [`05-Boundaries/`](../05-Boundaries/) | 知识库解释"为什么需要边界",实践层提供"具体边界规则" |
| runtime → Migration 概念 | [`06-Migration/`](../06-Migration/) | 知识库解释"迁移的本质",实践层提供"迁移步骤" |

**引用规则**:
- 知识库文档提及实践层概念时 → 链接到对应目录,一句话带过,不展开
- 实践层文档引用知识库概念时 → 链接回知识库对应文档,说明概念来源

## 9. 审查清单

发布前必须全部通过:

- [ ] 外行能看懂(无未解释的术语)
- [ ] 有类比或可视化(文本图 / 表格)
- [ ] 无可执行代码块(示意除外,且已标注"仅示意")
- [ ] 不超过 300 行
- [ ] 读完能用 3 句话复述核心
- [ ] 引用的实践层链接有效
- [ ] 元数据表格完整(层级=认知层)
- [ ] 无真实密钥 / 配置 / PII

**防孤岛检查**(见根目录 AGENTS.md § 6):

- [ ] 该文档已被 `00-目录索引` 或 `03-知识地图` 收录(入口 ≥ 1)
- [ ] 该文档末尾有"相关文件"链接,至少指向 1 篇关联文档(出口 ≥ 1,总览文档除外)
- [ ] 所有内部链接指向真实存在的文件(零死链)
- [ ] 若引用了其他文档,被引用文档已回引本文档(双向)
