# 模型选择与升级影响

## 一句话解释

模型选择决定 Agent 的推理能力、工具调用稳定性、成本、延迟和风险边界。

升级模型不是只换一个名字,它会改变 Agent 的行为风格、工具选择、输出格式和安全表现。

## 选择模型看什么

| 维度 | 说明 | 影响 |
|---|---|---|
| 推理能力 | 多步分析、复杂代码、抽象规划 | 决定能处理多复杂的任务 |
| 工具调用 | 是否稳定选择工具、参数是否准确 | 决定 Agent 能否可靠行动 |
| 长上下文 | 能否处理大量文件和历史 | 决定大型项目体验 |
| 指令遵循 | 是否稳定遵守边界和格式 | 决定可控性 |
| 速度 | 首 token 和总耗时 | 决定交互体验 |
| 成本 | 输入/输出/工具循环成本 | 决定能否规模化 |
| 多模态 | 图像、音频、视频理解 | 决定任务范围 |
| 代码能力 | 读写代码、测试、调试 | 决定开发型 Agent 效果 |
| 安全策略 | 拒答、敏感操作、合规能力 | 决定高风险场景可用性 |

## 模型不是越强越好

| 场景 | 推荐选择 |
|---|---|
| 简单问答、格式转换 | 快速低成本模型 |
| 代码实现、复杂排查 | 强推理 + 强工具调用模型 |
| 大量文档检索总结 | 长上下文 + 低成本输入模型 |
| 高风险自动化 | 指令遵循强 + 可审计 Runtime |
| 多 Agent 并行 | 主 Agent 用强模型,子任务可用便宜模型 |

强模型适合做判断和收敛,弱模型适合做批量、机械、低风险子任务。

## 升级模型前检查

升级前应记录旧模型基线:

- 常见任务成功率。
- 工具调用错误率。
- 输出格式稳定性。
- 平均成本和耗时。
- 边界规则遵守情况。
- 关键项目任务的回归样例。

没有基线就升级,很难判断“变好了”还是“只是风格变了”。

## 升级后重点观察

| 变化 | 可能影响 |
|---|---|
| 更主动 | 可能少问问题、直接行动 |
| 更谨慎 | 可能过度澄清、效率下降 |
| 更会调用工具 | 工具成本上升,权限风险增加 |
| 更少调用工具 | 可能凭记忆回答过期事实 |
| 输出更长 | 上下文和审阅成本增加 |
| 格式变化 | 自动化解析可能失败 |

## 模型与 Runtime 的关系

同一个模型在不同 Runtime 中表现可能不同,因为 Runtime 会影响:

- 系统提示词。
- 工具定义。
- 上下文压缩。
- 权限策略。
- 文件编辑方式。
- 错误恢复。
- 多 Agent 协作。

因此“模型评估”必须绑定 Runtime 和任务集,不能只看模型名称。

## 模型路由

生产系统常用模型路由:

| 路由方式 | 示例 |
|---|---|
| 按任务复杂度 | 简单分类用小模型,复杂规划用大模型 |
| 按风险 | 只读任务用快模型,写操作用稳模型 |
| 按阶段 | 规划用强模型,执行子步骤用快模型,评审再用强模型 |
| 按成本 | 批量摘要用便宜模型,最终决策用贵模型 |
| 按能力 | 图像任务路由到多模态模型 |

## 降级策略

模型不可用或成本过高时,Agent 应能降级:

- 从自动执行降级为给出步骤。
- 从全量分析降级为局部分析。
- 从复杂工具链降级为只读检索。
- 从长上下文降级为分批处理。
- 从生产操作降级为建议和审批草稿。
