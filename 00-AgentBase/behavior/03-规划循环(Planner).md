# Agent 规划循环(Planner)

> Agent 之所以能完成复杂任务,核心在于它有"规划-执行-反思"的循环——不是一次性输出结果,而是把任务拆解、逐步执行、根据反馈调整。理解规划循环,才能正确设计 Agent 的行为机制。

## 一句话理解

Agent 的规划循环就像一个"会自我纠错的工人"——接到任务后先拆解成几步,然后一步步执行,每步检查结果对不对,如果不对就调整方案再来,直到完成。

## 为什么需要规划循环

LLM 本身只是"输入→输出"的映射,没有"循环"和"反馈"。但复杂任务不能一步到位:

- **任务复杂**:搭建一个脚手架涉及十几个步骤,LLM 单次输出不可能完整规划
- **需要外部反馈**:要读取文件才知道里面写了什么,跑测试才知道代码对不对
- **需要纠错**:第一次方案不行,得根据失败信息调整
- **需要可观察**:用户/开发者要能看到 Agent 在做什么、做到哪了

规划循环就是把这些需求变成 Agent 的"工作模式"。

## 三种主流规划模式

```
┌──────────────────────────────────────────────────────────┐
│                  Agent 规划模式                             │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐  ┌───────────┐ │
│  │ ReAct          │  │ Plan-Execute   │  │ Reflection │ │
│  │ 边想边做         │  │ 先全规划再执行  │  │ 执行后反思 │ │
│  │                │  │                │  │           │ │
│  │ 思考→行动→观察  │  │ 规划→执行       │  │ 试错→总结 │ │
│  │ →再思考         │  │ →评估          │  │ →改进     │ │
│  └────────────────┘  └────────────────┘  └───────────┘ │
└──────────────────────────────────────────────────────────┘
```

### ReAct(Reasoning + Acting)

**特点**:每一步都"想清楚再行动",边做边调整。

```
任务: "查找 README.md 中提到的所有文件路径"

[思考]  用户想要找出 README 中提到的所有路径。
        我需要先读取 README 内容。
[行动]  file_read("README.md")
[观察]  内容包含 "src/main.ts" 和 "tests/test_main.ts" 两处路径。

[思考]  已经找到两处路径,可以汇报结果了。
[行动]  answer("README 中提到的路径: src/main.ts, tests/test_main.ts")
[观察]  任务完成。
```

**优点**:灵活、对环境变化响应快、适合探索性任务
**缺点**:频繁调用 LLM,成本高;长任务容易"迷失"

### Plan-and-Execute(先规划再执行)

**特点**:先把整个任务拆成完整计划,然后按计划执行,只在失败时重新规划。

```
[规划阶段]
任务: "搭建一个 Python CLI 工具"

计划:
1. 创建目录结构 (src/, tests/, docs/)
2. 写 pyproject.toml
3. 实现 src/cli.py
4. 写 tests/test_cli.py
5. 写 README.md
6. 运行 pytest 验证

[执行阶段]
[执行] step 1: mkdir 目录 → 成功
[执行] step 2: 写 pyproject.toml → 成功
[执行] step 3: 写 src/cli.py → 失败(pytest 报错)
[重规划] 修复 import 错误,重试 step 3 → 成功
[执行] step 4-6: 全部成功
```

**优点**:执行效率高(批量执行无需逐步询问 LLM)、计划可见
**缺点**:遇到意外需要重新规划、对环境适应性差

### Reflection(反思型)

**特点**:执行完后,让 Agent "复盘"自己的输出,发现问题就重做。

```
[生成]   写一篇文章 → 草稿 v1
[反思]   检查:有事实错误、段落衔接生硬
[改进]   修改事实,润色衔接 → 草稿 v2
[反思]   检查:逻辑通顺、字数达标 → 通过
[输出]   草稿 v2
```

**优点**:能显著提升输出质量(尤其写作/代码生成)
**缺点**:成本翻倍(每次都跑两轮)

## 各 Runtime 的规划循环实现

| Runtime | 默认模式 | 规划能力 | 反思能力 |
|---------|---------|---------|---------|
| Hermes Agent | ReAct | 完整 | 部分(通过 session 存储) |
| Claude Code | ReAct + Plan | 强(Plan Mode) | 无内置 |
| Codex CLI | ReAct | 部分 | 无内置 |

### Hermes Agent 的循环

Hermes 通过 CLI spinner + Gateway 进度推送,完整实现"思考→行动→观察"循环,详见 [Hermes 概述](../runtime/02-Hermes-Agent/01-概述.md)。

### Claude Code 的 Plan Mode

Claude Code 提供 `claude --plan` 进入计划模式——先生成完整计划,经用户确认后再执行,详见 [Claude Code 概述](../runtime/03-ClaudeCode/01-概述.md)。

### Codex CLI 的循环

Codex 默认采用 ReAct,通过 [Codex 安全与审批](../runtime/04-Codex-CLI/04-安全与审批.md) 控制每步动作是否需要审批。

## 规划循环的关键设计决策

### 决策 1:什么时候重新规划?

| 触发条件 | 行为 |
|----------|------|
| 步骤失败但可重试 | 重试当前步骤 |
| 步骤失败且不可重试 | 回滚到规划阶段 |
| 发现新信息需要调整后续步骤 | 部分重规划 |
| 所有步骤完成 | 任务结束 |

### 决策 2:上下文怎么管理?

长任务会撑爆上下文窗口,需要:

| 策略 | 说明 | 适用 |
|------|------|------|
| 滑动窗口 | 只保留最近 N 轮 | 短任务 |
| 摘要压缩 | 把历史压成摘要 | 中等任务 |
| 外部存储 | 把历史写到文件,按需读取 | 长任务 |

详见 [记忆机制](./04-记忆机制(Memory).md) 的"短期记忆"一节。

### 决策 3:何时需要用户介入?

| 介入点 | 触发条件 |
|--------|----------|
| 任务开始 | 任务模糊、目标不清 |
| 关键决策 | 高风险操作(删除、部署) |
| 异常恢复 | 步骤失败且 Agent 无法自动恢复 |
| 任务结束 | 汇报结果 |

## 规划与记忆的关系

```
┌────────────────────────────────────────────────────┐
│             Agent 规划循环                           │
│                                                    │
│  [任务输入]                                         │
│       ↓                                            │
│  [读取长期记忆]  ← 偏好、规则、踩坑经验              │
│       ↓                                            │
│  [读取实体记忆]  ← 相关人物、项目属性                │
│       ↓                                            │
│  [规划阶段]                                         │
│       ↓                                            │
│  [执行循环]                                         │
│    [思考] → 写入短期记忆(当前上下文)                 │
│    [行动] → 触发可观测日志                          │
│    [观察] → 评估结果                                │
│       ↓                                            │
│  [反思/总结]  → 写入长期记忆(新规则)                │
└────────────────────────────────────────────────────┘
```

详见 [记忆机制](./04-记忆机制(Memory).md)。

## 规划与可观测性/评估的关系

| 阶段 | 可观测性记录什么 | 评估度量什么 |
|------|-----------------|-------------|
| 规划 | 计划步骤、决策理由 | 计划合理性 |
| 执行 | 每步的输入输出、耗时 | 完成率、效率 |
| 反思 | 反思内容、改进点 | 反思有效性 |

详见 [可观测性](./06-可观测性(Observability).md) 和 [评估方法](./05-评估方法(Evaluation).md)。

## 规划循环的失败模式

| 失败模式 | 表现 | 缓解 |
|----------|------|------|
| 循环不收敛 | Agent 反复尝试同样错误 | 设置最大重试次数,强制升级 |
| 规划过大 | 一次规划太多步,中间迷失 | 限制单次规划步数,完成后重新规划 |
| 上下文溢出 | 长任务历史撑爆窗口 | 摘要压缩 + 外部记忆 |
| 缺乏反思 | 失败后不总结,下次还犯 | 强制反思步骤,写入长期记忆 |
| 过度规划 | 简单任务也走完整规划流程 | 根据任务复杂度选择模式 |
