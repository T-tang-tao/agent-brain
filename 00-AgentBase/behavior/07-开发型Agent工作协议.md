# 开发型 Agent 工作协议

## 一句话解释

开发型 Agent 工作协议是一套让 AI 编程助手在真实代码库中稳定工作的规则:先理解项目,再定位链路,只做最小改动,最后验证结果。

它适合专注开发的 Agent,尤其适合后端、前端、BFF、数据库脚本、接口契约、缓存、权限、并发等会互相影响的工程项目。

## 来源

本文从某个真实开发项目(代号 `oams-xj`)的项目级 `AGENTS.md`、前后端子项目规则和 `tmp_prompt_kit/` 提炼而来,所有具体路径、账号和环境地址已脱敏。

入库时已剥离具体业务名称、人员、账号、环境地址等项目特定信息,只保留可跨项目复用的开发方法。

## 核心分层

开发型 Agent 的规则不要堆在一个超长 prompt 里,应分三层维护:

| 层 | 放什么 | 例子 |
|---|---|---|
| 项目基线 | 长期稳定事实和边界 | 项目定位、技术栈、目录结构、启动命令、禁止修改区 |
| 任务提示词 | 单次任务执行方式 | 实现、修复、澄清、评审、排查 |
| 落地清单 | 新项目接入步骤 | 第 1 天补目录和命令,第 2 天补权限和隔离,第 3 天补排查经验 |

## 默认工作流

开发任务默认按以下顺序执行:

1. 定位需求所属模块。
2. 阅读入口、相邻实现、服务层、持久层、配置和测试。
3. 判断任务类型:修复、增强、重构、排查、评审。
4. 识别接口契约、数据结构、权限、缓存、并发、日志、测试等连带影响。
5. 复用项目已有模式做最小修改。
6. 运行相关验证。
7. 汇报变更内容、验证结果和剩余风险。

## 第一性原理检查

收到需求后先回答四个问题:

1. 真实目标是什么?
2. 当前已知事实是什么?
3. 缺少哪些关键上下文?
4. 最小可验证修改是什么?

如果目标不清、上下文不足、误改风险高,先澄清。不要用猜测补齐关键业务规则。

## 通用风险清单

开发型 Agent 在动代码前必须显式检查:

| 风险 | 检查点 |
|---|---|
| 接口契约 | 返回结构、错误码、字段类型、生成代码是否受影响 |
| 数据隔离 | 租户、站点、组织、权限范围是否遗漏 |
| SQL 安全 | 动态条件、排序、分页是否有注入风险 |
| 缓存 | key 是否包含隔离维度,写操作后是否失效 |
| 并发 | 幂等、锁粒度、事务边界、重复触发 |
| 审计 | 核心新增、删除、状态推进、批量操作是否记录日志 |
| 生成代码 | OpenAPI、路由树、客户端 SDK 等是否应通过命令生成 |
| 外部依赖 | 超时、重试、错误处理、配置项是否完整 |
| 数据库脚本 | 多数据库脚本版本号是否一致,占位脚本是否需要补齐 |
| 前端状态 | 服务端时间、权限可见性、表单禁用和提交中关闭等交互边界 |

## 后端项目常见规则

- 按 `controller/restservice -> service -> repository/dao -> sql/xml` 追调用链。
- SQL 优先放在统一持久层,不要散落在业务代码字符串中。
- 对外接口按业务隔离字段查询、更新、删除,不能只依赖全局唯一 ID。
- Service 层通过入参显式接收上下文,不要在深层自行读取默认上下文兜底。
- 缓存 key、锁 key、队列名、Topic 名应集中定义,不要在业务代码硬编码。
- 定时任务、后台扫描、队列消费必须考虑多副本、幂等和重复执行。
- 修改公共异常、鉴权白名单、全局契约前必须确认需求明确要求。

## 前端项目常见规则

- 自动生成的 service、路由树、类型文件不要手改,应通过项目命令重新生成。
- 优先复用现有组件、Hook、状态管理和设计系统。
- 当接口字段不存在时,优先反馈后端补字段,不要用不可靠的前端兜底伪造契约。
- 涉及当前时间时,优先使用项目封装的服务端时间工具。
- 表单提交中、弹窗关闭、权限可见性、低版本浏览器兼容性要纳入验证。
- 较长的 render 函数应拆成就近组件,保持路由页面清晰。

## 新项目接入清单

第 1 天必须完成:

- 明确项目名称、业务域、核心用户。
- 画出仓库目录和模块边界。
- 找到启动、测试、构建命令。
- 找到数据库、缓存、队列、外部依赖配置。
- 确认可改目录和禁止修改目录。
- 写出项目级 `AGENTS.md` 或等价规则文件。

第 2 天建议完成:

- 补齐接口契约、鉴权方式、日志审计方式。
- 补齐租户、站点、组织等数据隔离边界。
- 补齐缓存、并发、定时任务、分布式锁约束。
- 形成代码评审提示词。

第 3 天建议完成:

- 整理高频排查命令。
- 整理常见修改链路示例。
- 编写业务术语表。
- 列出易误改区域和高风险文件。

## 输出格式

开发型 Agent 完成任务后应至少说明:

- 变更内容
- 修改原因
- 验证结果
- 剩余风险

如果无法验证,必须说明原因和建议的后续验证方式。
