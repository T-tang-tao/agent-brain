# 可观测性

> Agent 不是黑盒——通过日志、追踪、监控和告警，让 Agent 的每次决策、每次工具调用、每笔成本都可审计、可回溯、可优化。

## 一句话理解

可观测性是给 Agent 装"行车记录仪"——不光记录最终结果，还记录"为什么这么做、调了什么工具、花了多少钱、哪里出了问题"。没有可观测性，Agent 出错时你只能猜。

## 为什么需要可观测性

| 没有可观测性 | 有可观测性 |
|-------------|-----------|
| "Agent 改错了文件，不知道为什么" | 追踪链显示第 3 轮推理时误判了文件路径 |
| "这个月 API 费用暴涨，不知道哪来的" | 成本监控显示某个 Skill 平均消耗 5 倍 token |
| "Agent 有时候很慢，不知道卡在哪" | 追踪显示 80% 耗时在等 MCP 服务器响应 |
| "上线后才发现 Agent 偶尔越界" | 告警在第一次越界时即触发 |
| "无法复盘 Agent 的决策过程" | 完整日志支持逐步回放每一轮推理 |

### 日志（Logs）

离散事件的记录——"什么时候发生了什么"。

| 日志类型 | 内容 | 示例 |
|----------|------|------|
| 推理日志 | 模型每轮的输入和输出 | `Round 3: input=..., output=tool_call(file_write, ...)` |
| 工具日志 | 每次工具调用的参数和结果 | `tool=shell, cmd="npm test", exit=0, duration=2.3s` |
| 审批日志 | 审批请求和用户响应 | `approval_requested: delete_file(x.js) → approved` |
| 沙箱日志 | 沙箱违规和拒绝 | `sandbox_violation: write to /etc/hosts → blocked` |
| 错误日志 | 异常和失败 | `error: MCP server timeout after 30s` |

> **原则**：日志要结构化（JSON），不要纯文本。结构化日志可以被程序解析和聚合。

### 追踪（Traces）

完整的执行链路——"从任务输入到完成，经过了哪些步骤"。

```
任务: "修复 auth.ts 中的 bug"
│
├─ Round 1: 推理
│  └─ 决策: 需要读取 auth.ts
│     └─ tool: file_read(auth.ts) ── 0.1s
│
├─ Round 2: 推理
│  └─ 决策: 搜索相关测试文件
│     └─ tool: search("auth.*test") ── 0.3s
│
├─ Round 3: 推理
│  └─ 决策: 修改 auth.ts 第 45 行
│     └─ tool: file_write(auth.ts, patch) ── 0.1s
│
├─ Round 4: 推理
│  └─ 决策: 运行测试验证
│     └─ tool: shell("npm test auth") ── 3.2s
│
└─ 完成: 4 轮推理, 4 次工具调用, 总耗时 3.7s
```

追踪的价值：

- **定位瓶颈**：哪个步骤最慢
- **发现冗余**：哪些工具调用是不必要的
- **回放复盘**：出错时逐步回放找到根因
- **对比优化**：修改前后对比执行链路变化

### 指标（Metrics）

聚合后的数值——"整体表现如何"。

| 指标类别 | 具体指标 | 说明 |
|----------|----------|------|
| 使用量 | 任务数、工具调用数、推理轮数 | 反映负载 |
| 性能 | 平均耗时、P95 耗时、工具调用延迟 | 反映速度 |
| 成本 | Token 消耗、API 费用、每任务成本 | 反映开销 |
| 质量 | 完成率、失败率、重试率 | 反映可靠性 |
| 安全 | 沙箱违规数、审批拒绝数、越界次数 | 反映安全性 |

> **与评估的区别**：指标是实时采集的运行数据（"现在 CPU 多少"），评估是周期性的系统性度量（"这个版本整体准确率多少"）。指标是评估的数据来源。

### 告警（Alerting）

基于日志、追踪和指标，在异常时主动通知。

| 告警类型 | 触发条件 | 通知方式 |
|----------|----------|----------|
| 成本告警 | 单任务成本超过阈值 | 通知 + 暂停 |
| 延迟告警 | P95 耗时超过阈值 | 通知 |
| 错误告警 | 错误率超过阈值 | 通知 + 告警 |
| 安全告警 | 沙箱违规或越界操作 | 通知 + 阻断 |
| 异常行为告警 | 工具调用模式异常（如突然大量删除） | 通知 + 阻断 |

## 各 Runtime 的可观测性

| Runtime | 日志 | 追踪 | 指标 | 告警 |
|---------|------|------|------|------|
| Hermes Agent | 完整（CLI spinner + Gateway 进度） | 会话存储（SQLite） | 部分 | 定时审计 |
| Claude Code | 对话历史 + 工具输出 | 对话回放 | 无内置 | 无 |
| Codex CLI | 会话日志 | 会话历史 | OTel 遥测（需配置） | 无内置 |

### Codex OTel 遥测

Codex CLI 支持通过 OpenTelemetry 发送遥测数据：

```toml
[otel]
endpoint = "http://localhost:4317"
service_name = "codex"
```

可采集指标包括工具调用次数/耗时、模型调用次数/token 消耗、沙箱违规事件、审批请求事件。

### Hermes 可观测执行

Hermes 的设计原则之一是"可观测执行"——每次工具调用对用户可见，CLI 有 spinner 显示执行状态，Gateway 有进度消息推送。

## 可观测性的层次

```
┌──────────────────────────────────────────┐
│  L3: 告警与自动化响应                      │
│  异常时通知 + 自动阻断/降级                 │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │  L2: 指标与仪表盘                   │  │
│  │  聚合数据可视化,趋势分析             │  │
│  │                                    │  │
│  │  ┌──────────────────────────────┐  │  │
│  │  │  L1: 追踪与链路分析            │  │  │
│  │  │  单次任务的完整执行链路         │  │  │
│  │  │                              │  │  │
│  │  │  ┌────────────────────────┐  │  │  │
│  │  │  │  L0: 日志               │  │  │  │
│  │  │  │  原始事件记录            │  │  │  │
│  │  │  └────────────────────────┘  │  │  │
│  │  └──────────────────────────────┘  │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

| 层次 | 回答什么 | 数据来源 |
|------|----------|----------|
| L0 日志 | 发生了什么？ | 原始事件 |
| L1 追踪 | 怎么发生的？ | 日志关联 |
| L2 指标 | 整体怎么样？ | 追踪聚合 |
| L3 告警 | 需要处理吗？ | 指标阈值 |

## 与评估的关系

可观测性是评估的数据来源：

```
可观测性（采集数据）──→ 评估体系（分析数据）──→ 优化改进
  日志                    正确性分析              修改 Prompt
  追踪                    效率分析                修改 Skill
  指标                    成本分析                调整配置
  告警                    稳定性分析              切换模型
```

详见 [Agent 评估体系](./05-评估方法(Evaluation).md)。

## 与边界的关系

可观测性是边界执行的审计手段：

- 沙箱违规 → 记录日志 → 触发告警
- 审批请求 → 记录日志 → 审计追溯
- 越界操作 → 告警 → 阻断 + 通知

详见 [Agent 边界与限制](../safety/01-Agent边界与限制.md)。

## Context 健康度信号(新增)

Context 退化是 Agent 表现降级最常见原因,需要专门的观测信号。

| 信号 | 检测方法 | 含义 |
  |---|---|---|
  | Context 占用率 | 监控 token 数 / 窗口大小 | > 70% 触发 compaction |
  | Lost-in-middle 症状 | 关键指令被忽略率 | > 10% 提示需要重新组织 context |
  | Poisoning 迹象 | 输出质量突然降级 + 工具调用错位 | 立即截断到污染点前 |
  | Tool output 膨胀 | 单次工具输出平均 token | > 2000 应自动落文件 |
  | Hallucination 率 | 与 ground truth 对比 | 持续 > X% 需重检 prompt |
  | Cross-task confusion | 多任务场景约束错位率 | > 5% 启用 sub-agent 隔离 |

**告警阈值参考**(按部署形态调整):

- Context 占用率 > 80%:立即 compaction
- Tool output 平均 > 3000 token:启用 observation masking
- Hallucination 率 > 5% 持续 1h:触发 prompt 审查
- Poisoning 检测:立即熔断,人工 review

更多 context 失败模式与对策见 [上下文工程总论](./09-Context-Engineering.md) § 4。
