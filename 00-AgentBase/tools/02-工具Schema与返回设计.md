# 工具 Schema 与返回设计

## 一句话解释

工具 schema 决定模型会不会选对工具,返回格式决定模型能不能正确使用工具结果。

工具不是“函数暴露出来就行”。对 Agent 来说,工具也是一份给模型读的接口文档。

## 工具设计目标

好工具应满足:

- 模型知道什么时候用。
- 参数少而明确。
- 错误可恢复。
- 返回可解析。
- 副作用可控。
- 权限边界清晰。

## 命名规则

| 规则 | 好例子 | 坏例子 |
|---|---|---|
| 动词开头 | `search_notes` | `notes` |
| 单一动作 | `create_task` | `manage_task` |
| 明确对象 | `read_project_file` | `read` |
| 避免同义重叠 | `list_threads` / `read_thread` | `get_thread` / `fetch_thread` 混用 |

名称要让模型一眼看出“动作 + 对象”。

## Description 写法

工具 description 至少包含四部分:

```text
做什么:
  用一句话说明工具能力。

Use when:
  列出触发场景。

Do not use when:
  列出容易误用的场景。

Returns:
  说明返回结构和关键字段。
```

不要只写“查询数据”“执行操作”这种泛描述。模型会依赖 description 做工具选择。

## 参数设计

| 原则 | 说明 |
|---|---|
| 少参数 | 参数越多,模型填错概率越高 |
| 强类型 | enum、number、boolean 优于自由文本 |
| 合理默认值 | 例如 `limit=20` |
| 明确单位 | 时间、金额、距离、页码必须写清 |
| 避免二义性 | `date_from/date_to` 优于 `date` |
| 不让模型填机密 | token、password 应由环境配置注入 |

## 返回格式

推荐返回结构:

```json
{
  "success": true,
  "data": {},
  "warnings": [],
  "next_actions": []
}
```

错误返回:

```json
{
  "success": false,
  "error_code": "NOT_FOUND",
  "message": "未找到目标文件",
  "recoverable": true,
  "suggested_action": "请检查路径或先搜索文件名"
}
```

## 返回内容原则

| 原则 | 说明 |
|---|---|
| 结构化优先 | JSON / 表格优于自由文本 |
| 保留来源 | 文件路径、URL、记录 ID、时间戳 |
| 控制体积 | 大结果分页、截断、摘要 |
| 显式不确定 | 区分无结果、权限不足、工具失败 |
| 给下一步 | 提供可操作建议 |

## 副作用分级

| 级别 | 例子 | 设计要求 |
|---|---|---|
| 只读 | 搜索、读取、查询 | 可默认允许 |
| 可逆写 | 创建草稿、写临时文件 | 返回回滚位置 |
| 持久写 | 修改文件、更新数据库 | 需要确认或白名单 |
| 外部影响 | 发消息、调用第三方 API | 需要审计和幂等 |
| 破坏性 | 删除、覆盖、支付、发版 | 默认禁止或强确认 |

## 工具错误设计

错误信息要帮助 Agent 恢复:

- 发生了什么。
- 哪个输入导致。
- 是否可重试。
- 是否需要换工具。
- 是否需要用户确认。

坏错误:

```text
error
```

好错误:

```text
未找到 threadId=abc 的任务。可能原因:任务不存在或当前工作区无权限。可先调用 list_threads 按标题搜索。
```

## 工具组合

单个工具不要过大。优先拆成:

- `list_*`: 找候选。
- `read_*`: 读详情。
- `create_*`: 新建。
- `update_*`: 修改。
- `delete_*`: 删除。

这样模型可以先观察再行动,也更容易设置权限。

## 测试工具

每个重要工具至少验证:

- 正常输入。
- 空结果。
- 无权限。
- 参数错误。
- 超时或外部服务失败。
- 大结果分页或截断。
- 重复调用是否幂等。
