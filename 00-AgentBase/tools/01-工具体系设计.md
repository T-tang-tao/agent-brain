# 工具体系设计

> 本文讲解 Agent 工具体系的设计原则、三类工具的边界、schema 设计最佳实践。
>
> 概念辨析见根级 [`../02-概念全景辨析.md`](../02-概念全景辨析.md)。

## 1. Tool 是什么

**Tool = Agent 可调用的动作**。模型输出"调用工具 X 的参数 Y",Runtime 执行实际动作。

工具调用示例(OpenAI 风格):

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "杭州"
  }
}
```

## 2. 三类工具的边界

| 类型 | 定义 | 优先级 | 例子 |
|------|------|--------|------|
| **Built-in Tools** | Runtime 内置 | Runtime 提供 | Read/Write/Bash/Grep/Glob |
| **MCP Tools** | 通过 MCP 协议接入 | 中 | 数据库查询、API 调用、文件搜索 |
| **Custom Tools** | 用户自定义 | 最低 | 项目特定的 helper 函数 |

### 2.1 Built-in Tools(首选)

- **优势**:Runtime 深度优化、权限可控、用户熟悉
- **限制**:能力受限于 Runtime 设计
- **何时用**:Agent 90% 的工具调用应该是这类

### 2.2 MCP Tools(连接外部)

- **优势**:协议统一、可复用、生态丰富
- **限制**:有通信开销、需要配置 MCP server
- **何时用**:需要连接数据库、外部 API、第三方服务时

**MCP 知识**:见 [../runtime/01-Agent运行时与CLI.md](../runtime/01-Agent运行时与CLI.md) (各 Runtime 的 MCP 集成章节)

### 2.3 Custom Tools(项目特定)

- **优势**:完全可控
- **限制**:需自己编写、调试、维护
- **何时用**:项目有非常特定的功能,无法用 Built-in 或 MCP 实现

**项目级 Custom Tools 应该放在**:`.agent/skills/<skill>/scripts/` 或 `.agent/tools/`

## 3. 工具 schema 设计最佳实践

### 3.1 命名

| 原则 | 好 | 坏 |
|------|-----|-----|
| 动词开头 | `get_weather` | `weather` |
| snake_case | `read_file` | `readFile` |
| 描述性强 | `search_documents` | `search` |

### 3.2 描述(docstring)

**关键**:模型的工具选择决策**100% 依赖 description**。

```python
@function_tool
def search_documents(query: str, max_results: int = 10) -> str:
    """在本地知识库中搜索文档。

    Use this when... 用户要"查找 X 文档"、"在 Y 中搜索 Z"、"找关于 X 的资料"。
    Do NOT use when... 用户要"创建新文档"(用 create_document)、"看某个特定文件"(用 read_file)。
    
    Args:
        query: 搜索关键词(支持中文)
        max_results: 最大返回数量,默认 10
    """
```

**好的 description**:
- 说明工具做什么
- 说明何时用(Use this when...)
- 说明何时不用(Do NOT use when...)
- 关键参数解释

### 3.3 返回值

| 原则 | 说明 |
|------|------|
| **结构化优先** | 返回 JSON/字典而非自由文本,便于模型处理 |
| **错误显式** | 返回包含 `success: false` + `error: "..."` |
| **截断大输出** | 超过 ~2000 token 返回截断+摘要 |
| **避免副作用** | 工具应该幂等(同样输入同样输出) |

### 3.4 错误处理

```python
def safe_tool(input: str) -> dict:
    try:
        result = risky_operation(input)
        return {"success": True, "data": result}
    except Exception as e:
        return {"success": False, "error": str(e), "input": input}
```

**Agent 应该能从错误中恢复**:错误信息要可操作。

## 4. 工具与 Skill 的关系

| 维度 | 工具(Tool) | 技能(Skill) |
|------|------------|--------------|
| **粒度** | 单个动作 | 完整工作流 |
| **入口** | 模型直接调用 | 通过 SKILL.md 描述触发 |
| **实现** | Python 函数 | 文件夹(SKILL.md + scripts + references) |
| **复用** | Runtime 内通用 | 跨 Runtime 通过分发 |

**关系**:Skill 内部**使用**多个 Tool 完成工作流。

详见 [Skill 设计](../behavior/01-如何编写Skill.md) 和 [Tool 辨析](../02-概念全景辨析.md)

## 5. 工具权限分级

按危险程度分级:

| 级别 | 工具类型 | 是否需要确认 |
|------|----------|--------------|
| L0 只读 | Read/Grep/Glob | 否 |
| L1 网络 | WebFetch/WebSearch | 通常否 |
| L2 写文件 | Write/Edit | 默认是,白名单除外 |
| L3 Shell | Bash | 默认是 |
| L4 破坏性 | rm/drop/delete | 总是需要确认 |

**配置**:各 Runtime 不同
- Claude Code:`permissions.allow` / `permissions.deny`
- Codex:审批策略
- Hermes:见 [../safety/01-Agent边界与限制.md](../safety/01-Agent边界与限制.md)
