# Agent 是什么 — 人类入门

> 这篇文章帮人类搞懂 Agent 到底是什么。如果你已经理解 Agent、模型、Workflow 的区别，可以跳过。

## 从一个例子开始

假设你需要修复一个 bug。

**如果你问一个普通 AI 模型**，它可能会告诉你："你应该检查 xxx 文件，可能是 yyy 的问题，建议修改 zzz。"——它只**回答**了你的问题。

**如果你让一个 Agent 去做**，它会：

1. 读取你描述的 bug
2. 搜索代码库里相关文件
3. 阅读关键代码，理解上下文
4. 找到问题，修改代码
5. 运行测试
6. 如果测试还失败，继续定位
7. 最终告诉你：改了什么、为什么、怎么验证的

区别在于：模型在**回答**，Agent 在**行动**。

## Agent 的一句话定义

Agent 是一个"会使用工具完成任务的模型运行系统"。

换句话说：

```text
Agent = Model + Goal + Context + Tools + Runtime + Memory + Permission + Verification
```

模型只是 Agent 的"大脑"。Agent 还需要工具、上下文、运行时、权限、记忆和验证机制，才能真正完成任务。

## Agent 和模型的区别

很多人会把 Agent 和模型混在一起，但它们不是一回事。

| 概念 | 作用 |
|------|------|
| Model | 负责理解、推理、生成内容 |
| Agent | 使用模型做决策，并调用工具完成任务 |

再举个例子：

```text
用户：帮我看看这个项目有没有安全问题。

普通模型：可能会告诉你一般的安全检查清单。
Agent：会实际读取代码、搜索漏洞模式、检查依赖、运行安全扫描、汇总发现的问题。
```

模型主要在"回答"；Agent 可以在环境中"行动"。

## Agent 的组成

一个实用的 Agent 通常包括这些部分：

| 组成 | 说明 | 生活类比 |
|------|------|----------|
| Model | 推理和生成能力（Claude、GPT 等） | 大脑 |
| Goal | 当前任务目标 | 任务单 |
| Context | 对话历史、项目文件、用户要求 | 工作台上的资料 |
| Tools | 可调用的动作（读文件、跑命令、查网页） | 手和工具箱 |
| Runtime | 执行循环和宿主环境（Claude Code、Codex CLI 等） | 办公室 |
| Memory | 跨会话保存的长期信息 | 笔记本 |
| Permission | 权限和确认机制 | 门禁卡 |
| Verification | 验证机制（测试、lint、人工确认） | 质检员 |

## 为什么 Agent 能"做事"

Agent 的关键在于它有一个**循环**：

```text
观察当前状态 → 判断下一步 → 调用工具 → 读取结果 → 继续判断 → 完成或停止
```

这不是预先写死每一步的流程，而是 Agent 根据当前状态**动态决定**下一步做什么。这让 Agent 能处理开放式任务，而不是只生成一次回答。

## Agent、模型、Workflow 有什么区别

| 概念 | 本质 | 是否动态决策 | 适合场景 |
|------|------|:-----------:|----------|
| Model | 推理和生成能力 | 否 | 作为底层智能能力 |
| LLM Call | 一次输入输出 | 否 | 总结、分类、翻译 |
| Workflow | 固定步骤自动化 | 否 | 稳定、可预定义的流程 |
| Agent | 模型驱动的动态执行循环 | 是 | 开放式、多步骤、不确定任务 |

### Workflow 和 Agent 的关键区别

Workflow 的步骤是**预先定义的**，由程序逻辑控制。比如：

```text
读取文件 → 提取标题 → 调用模型总结 → 保存结果
```

Agent 的步骤是**动态决定的**，由模型根据观察结果决定。比如：

```text
用户：修复测试失败。
Agent：
1. 先运行测试（观察）
2. 根据失败日志搜索代码（判断 + 行动）
3. 修改可能原因（行动）
4. 再跑测试（验证）
5. 如果还有失败，继续定位（循环）
6. 直到完成或遇到阻塞
```

### 选择原则

```text
如果下一步是固定的，用 Workflow。
如果下一步要根据观察结果决定，用 Agent。
```

### 什么时候不需要 Agent

- 单次总结
- 简单分类
- 固定格式抽取
- 规则明确的批处理
- 已经能用普通脚本稳定完成的任务

这些情况用 LLM Call 或 Workflow 更简单、更便宜、更可控。

### 什么时候适合 Agent

- 任务有多个步骤，但无法提前完全写死
- 需要读文件、搜索、修改、验证
- 需要根据工具结果调整下一步
- 需要长时间探索
- 需要处理不完整需求
- 需要跨工具协作

例如：修复 bug、迁移项目、生成并验证报告、分析大型代码库、多来源调研。
