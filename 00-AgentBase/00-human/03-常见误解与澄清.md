# 常见误解与澄清

> 这篇文章收集了关于 Agent 和知识库的常见误解。每条先说误解，再说为什么不对。

## 关于 Agent 的误解

### 误解 1：只要用了大模型就是 Agent

**不是。** 单次问答或总结通常只是 LLM 调用，不一定是 Agent。

判断标准：有没有**动态决策循环**？如果只是"输入 → 输出"一次完成，那是 LLM Call。如果根据工具结果调整下一步，那才是 Agent。

### 误解 2：Agent 一定完全自主

**不是。** 好的 Agent 通常需要边界、权限和人工确认。

越能行动的 Agent，越需要限制。就像越强的工具，越需要安全防护——电锯比手工锯强，但也更危险。

### 误解 3：Agent 越强越不需要知识库

**不是。** 强模型仍然需要正确上下文、实时数据、项目知识和验证机制。

模型再强，也不知道你项目的目录结构、你团队的技术选型、你上个月踩了什么坑。知识库不是给模型"补脑"的，是给 Agent 提供**它无法从训练数据中获得的信息**。

### 误解 4：Agent 就是 Workflow

**不是。** Workflow 是固定流程；Agent 是模型根据状态动态决定下一步。

| 维度 | Workflow | Agent |
|------|----------|-------|
| 步骤 | 预先定义 | 动态决定 |
| 控制者 | 程序逻辑 | 模型 + 运行时 |
| 适合任务 | 稳定流程 | 开放式任务 |
| 可预测性 | 高 | 较低，需要边界和验证 |

## 关于知识库的误解

### 误解 5：知识库就是向量数据库

**不是。** 向量数据库只是检索实现之一。

知识库是人和 Agent 共同读取的知识系统，由 wiki、文档、RAG、Memory、Skill、Prompt、项目资料和实时工具共同组成。向量数据库可以服务于知识库，但知识库远不止于此。

### 误解 6：所有内容都应该被 Agent 记住

**不是。** 临时信息、敏感信息、可从代码中重新获得的信息，不应随便写入长期记忆。

Memory 应该**小而准**——只保存未来有用的长期信息。就像你的笔记本不会记录今天午饭吃了什么，但会记下"这个项目不能用 var 声明变量"。

### 误解 7：文档越多越好

**不是。** 过期、重复、矛盾的文档会让 Agent 更混乱。

知识库的价值在于**质量**，不在于数量。一篇准确的、有交叉引用的文档，比十篇散落、过时、矛盾的文档有用得多。

## 关于边界的误解

### 误解 8：Agent 能做的事越多越好

**不是。** 能力越大，风险越大。

Agent 可以调用工具，风险会从"说错"扩大到"做错"：改错文件、删除数据、执行危险命令、调用外部 API 产生真实世界影响。

所以 Agent 不能只靠"聪明"，还需要**权限分级**和**确认机制**：
- 只读操作可以自主
- 写入操作需要在任务范围内
- 高风险操作（删除、推送、部署、交易）必须确认

### 误解 9：Agent 完成任务说"应该可以"就够了

**不是。** Agent 完成任务后应该**验证**，而不是只说"应该可以"。

验证包括：测试是否通过、lint/typecheck 是否通过、输出文件是否存在、引用来源是否真实。

如果无法验证，应该如实说明："已完成修改，但没有运行测试，因为当前环境缺少依赖。"
