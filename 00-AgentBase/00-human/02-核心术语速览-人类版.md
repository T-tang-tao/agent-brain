# 核心术语速览 — 人类版

> 这篇文章帮人类快速搞懂 Agent 体系里的关键术语。每个术语配一句话定义 + 生活类比 + 例子。

## 为什么这些术语容易混淆

因为它们都和"让 Agent 变强"有关，但解决的问题不同：

- Tool 让 Agent **能做事**
- MCP 让 Agent **能连接外部系统**
- Skill 让 Agent **知道某类任务怎么做**
- Plugin 让宿主系统**获得新能力**
- Prompt 规定 Agent **当前怎么行动**
- Knowledge Base 提供 Agent **可以查的资料**
- Memory 让 Agent **记住长期经验**
- RAG 解决**如何从大量资料中找出当前相关内容**

不区分的话，后续理解会很混乱。

## 速览表

| 术语 | 一句话 | 生活类比 | 例子 |
|------|--------|----------|------|
| Tool | Agent 能调用的动作 | 手 | 读文件、跑命令、查网页 |
| MCP | 连接外部工具的标准接口 | 万能插座 | 让 Agent 连接 GitHub、数据库 |
| Skill | 某类任务的操作流程 | 技能证书 | "调试 bug 要先复现再定位再修复" |
| Plugin | 宿主系统的扩展包 | 插件扩展 | 给编辑器安装新功能包 |
| Prompt | 给模型的行为指令 | 工作指令 | "你是一个 Python 专家，请…" |
| Knowledge Base | Agent 可查询的资料 | 图书馆 | wiki、文档、API 说明 |
| Memory | 跨会话的长期记忆 | 笔记本 | 用户偏好、项目约束、历史踩坑 |
| RAG | 从大量资料中检索相关内容 | 图书管理员帮你找书 | 用户问题 → 检索文档 → 注入上下文 → 回答 |

## 逐个解释

### Tool — Agent 能做什么动作

Tool 是 Agent 可以调用的动作。比如：

- 读取文件
- 写入文件
- 搜索代码
- 执行 shell 命令
- 查询数据库
- 调用行情 API
- 创建 GitHub issue

> **Tool 解决"Agent 能做什么动作"。**

### MCP — 如何标准化连接外部系统

MCP（Model Context Protocol）是 Agent 连接外部工具和资源的**标准接口**。

MCP server 可以暴露：
- tools（可调用动作）
- resources（可读取资源）
- prompts（可复用提示模板）

> **MCP 解决"Agent 如何标准化连接外部系统"。**

MCP 不是工具本身，而是连接工具和资源的**协议**。就像 USB 不是设备，而是连接设备的标准接口。

### Skill — 某类任务应该怎么做

Skill 是某类任务的**操作流程、上下文和最佳实践**。

比如：
- 调试 bug 应该先复现，再定位，再修复，再验证
- code review 应该先看 correctness，再看简化、效率和测试
- 数据可视化应该先判断图表类型，再设计颜色、坐标轴和标注

> **Skill 解决"这类任务应该怎么做"。**

Skill 不一定执行动作，它主要提供**方法和流程**。就像一本烹饪教程，告诉你做这道菜的标准步骤。

### Plugin — 如何扩展宿主系统

Plugin 是宿主系统的**扩展机制**。

某个 agent runtime 允许安装 plugin，plugin 可能提供：
- 新命令
- 新 UI
- 新工具
- 新配置
- 新集成
- 新 skill 包

> **Plugin 解决"如何扩展宿主系统能力"。**

Plugin 可能包含 tool、skill、prompt 或配置，但它本身是**扩展包**的概念。就像浏览器插件可以包含新功能、新工具栏、新设置。

### Prompt — Agent 当前如何理解任务

Prompt 是给模型或 Agent 的**行为指令**。

常见 prompt 包括：
- system prompt（系统指令）
- developer prompt（开发者指令）
- task prompt（任务指令）
- agent role prompt（角色指令）
- boundary prompt（边界指令）

> **Prompt 解决"Agent 当前应该如何理解任务和约束"。**

Prompt 可以告诉 Agent 做什么、不要做什么、输出什么格式、什么时候调用工具。

### Knowledge Base — Agent 可以查什么资料

Knowledge Base 是 Agent 可以查询的**资料集合**。

比如：
- wiki
- 文档
- API 说明
- 项目架构笔记
- 策略说明
- FAQ
- 历史研究记录

> **Knowledge Base 解决"Agent 可以查什么资料"。**

### Memory — Agent 长期记住什么

Memory 是 Agent 跨会话保存的**长期信息**。

比如：
- 用户偏好
- 项目约束
- 已确认的方案
- 历史踩坑
- 下次应该应用的经验

> **Memory 解决"Agent 以后还应该记住什么"。**

Memory 应该**小、准、可维护**，不应该变成垃圾堆。就像你的笔记本只记重要的事，不会把每句话都写进去。

### RAG — 如何检索相关知识

RAG（Retrieval-Augmented Generation）是**检索增强生成**。

典型流程：

```text
用户问题 → 检索相关文档 → 注入上下文 → 模型回答
```

> **RAG 解决"如何从大量资料中找出当前相关内容"。**

RAG 是知识库的一种**使用方式**，不等于所有知识库。

## 它们怎么配合

一个成熟 Agent 系统大致是这样运作的：

```text
Prompt 定义当前行为     ← "你现在是个代码审查员"
Skill 提供任务流程       ← "代码审查的步骤是…"
Knowledge Base 提供资料  ← "项目架构文档在这里"
Memory 提供长期经验      ← "上次审查发现的问题"
MCP 连接外部系统         ← "连接 GitHub 仓库"
Tool 执行动作            ← "读取文件、运行测试"
Plugin 扩展宿主能力      ← "安装了代码分析插件"
Runtime 管理循环和权限   ← "在 Claude Code 里运行"
Model 负责推理和决策     ← "由 Claude 来做判断"
```

## 重要边界

这些概念虽然相关，但不是一回事：

| 边界 | 说明 |
|------|------|
| Skill 不是 MCP | Skill 是流程知识，MCP 是连接协议 |
| MCP 不是知识库 | MCP 可以暴露知识资源，但 MCP 本身不是知识库 |
| Plugin 不是 Prompt | Plugin 是扩展包，Prompt 是指令文本 |
| Memory 不是 RAG | Memory 是长期记忆，RAG 是检索方式 |
| Tool 不是 Agent | Tool 是 Agent 调用的动作，Agent 是决策和执行循环 |
