## 双层记忆机制

Claude Code 的记忆由两个互补系统构成,一个由你显式编写,一个由 Claude 自动积累:

| 维度 | CLAUDE.md 文件 | Auto memory |
|------|----------------|-------------|
| **谁写** | 你手动编写 | Claude 自动学习 |
| **包含什么** | 项目指令、编码标准、架构决策 | 纠正、构建命令、调试经验、用户偏好 |
| **作用域** | 文件系统层级(系统/用户/项目/本地) | 按仓库存储(per-repository) |
| **加载位置** | 每次会话注入系统提示 | 每次会话注入前 200 行 / 25KB |
| **用于** | 你想始终生效的规则 | Claude 从交互中学到的东西 |

> 类比:CLAUDE.md 像"团队手册"——你写好规章制度人人遵守;Auto memory 像"老员工的工作笔记"——Claude 自己摸索积累的经验。两者互补,缺一不可。

## CLAUDE.md 加载位置

CLAUDE.md 从最宽到最窄逐级加载,所有发现的文件拼接(不是覆盖):

| 优先级 | 位置 | 作用域 | 说明 |
|--------|------|--------|------|
| 1(最高) | Managed policy 系统路径 | 企业级 | 由管理员通过策略文件强制下发 |
| 2 | `~/.claude/CLAUDE.md` | 用户级 | 跨所有项目的个人偏好 |
| 3 | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 项目级 | 团队共享,提交到 git |
| 4 | `./CLAUDE.local.md` | 本地级 | 个人项目覆盖,不提交 |

> 类比:像 CSS 的层叠规则——企业策略 > 用户偏好 > 项目约定 > 本地覆盖,从宽到窄逐级叠加。

## 加载机制

CLAUDE.md 不是单一文件覆盖,而是**目录树向上遍历拼接**:

- 从当前工作目录(cwd)向上逐级查找 CLAUDE.md
- 所有发现的文件按 **根目录 → 工作目录** 顺序拼接(不是后者覆盖前者)
- 子目录的 CLAUDE.md **按需加载**——当 Claude 读取该子目录的文件时才加载

> 类比:像 Git 的 `.gitignore`——从仓库根到当前目录的规则全部生效,层层叠加而非互相覆盖。

## @import 导入语法

在 CLAUDE.md 中用 `@` 前缀导入其他文件内容:

```markdown
@path/to/file           # 导入相对路径文件
@/absolute/path/file    # 导入绝对路径文件
@AGENTS.md              # 导入 AGENTS.md 实现兼容
```

导入规则:

| 规则 | 说明 |
|------|------|
| 时机 | 启动时一次性导入 |
| 路径 | 支持相对路径和绝对路径 |
| 深度 | 最多 4 跳(导入链不能超过 4 层) |
| 转义 | 用反引号包裹 `@path` 可阻止导入(显示为字面文本) |

## AGENTS.md 兼容性

Claude Code **只读 CLAUDE.md,不读 AGENTS.md**。但可以通过 import 实现兼容:

```markdown
# CLAUDE.md
@AGENTS.md
```

在 CLAUDE.md 顶部加一行 `@AGENTS.md`,即可让 Claude Code 加载 AGENTS.md 的全部内容。这让支持多 runtime 的项目只需维护一份 AGENTS.md。

> 类比:像适配器模式——CLAUDE.md 是 Claude Code 的原生接口,`@AGENTS.md` 是转接头,让通用标准接入。

## /init 命令

`/init` 通过分析代码库自动生成 CLAUDE.md:

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| 默认模式 | 直接运行 `/init` | 单次分析代码库,生成 CLAUDE.md |
| 交互多阶段模式 | 设置 `CLAUDE_CODE_NEW_INIT=1` | 多阶段交互流程,逐步确认生成内容 |

生成的 CLAUDE.md 包含项目结构、构建命令、编码约定等,可作为团队协作的起点。

## .claude/rules/ 目录

对于大型项目,可将规则拆分为模块化文件:

```
.claude/rules/
├── coding-style.md      # 编码风格
├── api-conventions.md   # API 约定
└── testing.md            # 测试规范
```

每个规则文件支持 YAML frontmatter 的 `paths` 字段,实现**路径特定规则**:

```yaml
---
paths:
  - "src/api/**/*.ts"    # glob 模式
---
API 层必须使用 Zod 验证所有输入...
```

只有当 Claude 操作匹配路径的文件时,该规则才被加载。

> 类比:像路由守卫——规则不全部常驻,只在进入对应"路由"(文件路径)时才激活。

## 编写有效指令

CLAUDE.md 的质量直接影响 Claude 的行为:

| 做 | 不做 |
|----|------|
| 控制在 200 行以内 | 堆砌冗长描述 |
| 用标题和列表组织 | 写大段散文 |
| 写具体可执行的指令 | 写模糊的原则 |
| 保持指令一致性 | 前后矛盾 |
| 用命令而非描述 | "你应该小心" → "运行 `npm test`" |

## Auto memory

Claude 自动从交互中学习并持久化经验:

| 学习来源 | 保存内容 |
|----------|----------|
| 用户的纠正 | "别用 var,用 const" |
| 构建命令 | `npm run build:prod` 的正确参数 |
| 调试经验 | 某个报错的根因和修复方法 |
| 用户偏好 | "提交信息用中文" |

- 按仓库存储(per-repository),每个项目独立
- 每次会话自动加载前 200 行或 25KB
- 无需手动编写,Claude 自主管理

## /memory 命令

在会话中用 `/memory` 查看和编辑记忆:

- 查看当前加载的所有 CLAUDE.md 文件
- 查看和编辑 Auto memory 内容
- 直接在编辑器中修改

## Monorepo 排除

大型 monorepo 中,可用 `claudeMdExcludes` 排除特定子项目的 CLAUDE.md:

```json
{
  "claudeMdExcludes": ["submodules/legacy/", "vendor/"]
}
```

避免无关子项目的指令污染当前上下文。

> 类比:像 `.gitignore`——主动声明哪些记忆文件不加载,保持上下文干净。

## HTML 注释处理

CLAUDE.md 中的 HTML 注释(`<!-- -->`)在加载到上下文前会被**剥离**。这让你可以在源文件中留笔记给自己,而不会占用 Claude 的 token 预算。

## 配置参考

| 配置项 | 作用 |
|--------|------|
| `CLAUDE_CODE_NEW_INIT=1` | 启用交互式多阶段 /init 流程 |
| `claudeMdExcludes` | 排除特定路径的 CLAUDE.md |
| `.claude/rules/` | 模块化路径特定规则 |
| `@import` | 在 CLAUDE.md 中导入其他文件 |
