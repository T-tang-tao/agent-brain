## 配置作用域

Claude Code 使用作用域系统决定配置的应用范围和共享对象:

| 作用域 | 位置 | 影响范围 | 团队共享 |
|--------|------|----------|----------|
| **Managed**(托管) | Server-managed / MDM / 系统级 `managed-settings.json` | 组织全员或本机所有用户 | 是(IT 部署) |
| **User**(用户) | `~/.claude/` 目录 | 你,跨所有项目 | 否 |
| **Project**(项目) | 仓库内 `.claude/` | 本仓库所有协作者 | 是(提交到 git) |
| **Local**(本地) | `.claude/settings.local.json` | 仅你,仅本仓库 | 否(gitignore) |

> **类比**:Managed 像"公司政策不可违抗",User 像"个人偏好全局生效",Project 像"团队约定提交到仓库",Local 像"个人草稿不提交"。

### 各特性使用的作用域

| 特性 | User 位置 | Project 位置 | Local 位置 |
|------|-----------|--------------|------------|
| **Settings** | `~/.claude/settings.json` | `.claude/settings.json` | `.claude/settings.local.json` |
| **Subagents** | `~/.claude/agents/` | `.claude/agents/` | 无 |
| **MCP servers** | `~/.claude.json` | `.mcp.json` | `~/.claude.json`(per-project) |
| **Plugins** | `~/.claude/settings.json` | `.claude/settings.json` | `.claude/settings.local.json` |
| **CLAUDE.md** | `~/.claude/CLAUDE.md` | `CLAUDE.md` / `.claude/CLAUDE.md` | `CLAUDE.local.md` |

> Windows 上 `~/.claude` 解析为 `%USERPROFILE%\.claude`。

## settings.json 文件

`settings.json` 是 Claude Code 的官方配置机制,采用分层结构:

| 文件 | 作用 |
|------|------|
| `~/.claude/settings.json` | 用户设置,跨所有项目生效 |
| `.claude/settings.json` | 项目设置,提交到 git 与团队共享 |
| `.claude/settings.local.json` | 本地设置,不提交,用于个人偏好和实验 |

### 示例

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test *)", "Read(~/.zshrc)"],
    "deny": ["Bash(curl *)", "Read(./.env)", "Read(./secrets/**)"]
  },
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1"
  }
}
```

> `$schema` 行启用 VS Code / Cursor 中的自动补全和内联校验。

## 托管设置

组织需要集中控制时,Claude Code 支持三种托管设置投递机制:

| 机制 | 投递方式 | 说明 |
|------|----------|------|
| **Server-managed** | 登录时远程投递 | 从 Anthropic 服务器或自托管 Claude apps gateway 推送 |
| **MDM / OS 策略** | 原生设备管理 | macOS: `com.anthropic.claudecode` plist;Windows: `HKLM\SOFTWARE\Policies\ClaudeCode` 注册表 |
| **File-based** | 系统目录文件 | macOS: `/Library/Application Support/ClaudeCode/`;Linux: `/etc/claude-code/`;Windows: `C:\Program Files\ClaudeCode\` |

### MDM 部署位置

| 平台 | 位置 |
|------|------|
| macOS | plist 域 `com.anthropic.claudecode`(通过 Jamf / Kandji 等部署) |
| Windows(管理员级) | `HKLM\SOFTWARE\Policies\ClaudeCode` 的 `Settings` 值(REG_SZ,含 JSON) |
| Windows(用户级) | `HKCU\SOFTWARE\Policies\ClaudeCode`(最低策略优先级) |

## 配置优先级

同一设置出现在多个作用域时,按以下优先级从高到低生效:

```
1. Managed(托管)          ← 最高,不可被覆盖
2. CLI 参数                 ← 临时会话覆盖
3. Local(本地)             ← 覆盖 Project 和 User
4. Project(项目)            ← 覆盖 User
5. User(用户)              ← 最低,无其他指定时生效
```

> **类比**:像"法律 > 公司规定 > 部门约定 > 个人偏好"——高优先级覆盖低优先级,托管设置如法律不可违抗。

**注意**:权限规则(`allow`/`deny`)跨作用域**合并**而非覆盖,详见权限与安全。

## 关键设置项

| 设置键 | 作用 |
|--------|------|
| `permissions` | 权限规则(allow / ask / deny 数组) |
| `env` | 环境变量(如 `CLAUDE_CODE_GIT_BASH_PATH`、`DISABLE_AUTOUPDATER`) |
| `autoUpdatesChannel` | 发布渠道:`latest`(默认)或 `stable` |
| `model` | 使用的模型 |
| `defaultMode` | 默认权限模式 |
| `hooks` | 生命周期钩子配置 |
| `minimumVersion` | 最低版本锁定 |

## 权限规则语法

权限规则统一使用 `ToolName(specifier)` 格式:

| 规则格式 | 作用于 | 示例 |
|----------|--------|------|
| `Bash(npm run *)` | Bash、Monitor | 命令模式匹配 |
| `Read(~/secrets/**)` | Read、Grep、Glob、LSP | 路径模式匹配 |
| `Edit(/src/**)` | Edit、Write、NotebookEdit | 路径模式匹配 |
| `Skill(deploy *)` | Skill | 技能名称匹配 |
| `Agent(Explore)` | Agent | 子代理类型匹配 |
| `WebFetch(domain:example.com)` | WebFetch | 域名匹配 |
| `WebSearch` | WebSearch | 无 specifier,整体允许/拒绝 |

> `Edit(...)` 允许规则同时授予同路径的读访问,无需额外配 `Read(...)`。

## 环境变量

通过 `env` 键设置常用环境变量:

| 变量 | 作用 |
|------|------|
| `CLAUDE_CODE_GIT_BASH_PATH` | Git Bash 路径(Windows) |
| `DISABLE_AUTOUPDATER` | 设为 `1` 禁用后台自动更新 |
| `MCP_TIMEOUT` | MCP 服务器连接超时 |
| `MAX_MCP_OUTPUT_TOKENS` | MCP 输出 token 上限 |
| `USE_BUILTIN_RIPGREP` | 设为 `0` 禁用内置 ripgrep(Alpine) |
| `CLAUDE_CODE_USE_POWERSHELL_TOOL` | 设为 `1` 启用 PowerShell 工具 |

## 生效时机

| 行为 | 说明 |
|------|------|
| **热重载** | Claude Code 监视 settings 文件,变更即生效(含 permissions、hooks、credential helpers) |
| **需重启** | `model`(用 `/model` 切换)和 `outputStyle`(系统提示词重建)在会话启动时读取 |

## 托管设置容错

托管设置采用**容错解析**:

| 场景 | 行为 |
|------|------|
| 普通无效条目 | 剥离该条目,记录警告,强制执行其余有效策略 |
| 安全字段无效 | 逐字段处理(fail-safe) |
| `allowedMcpServers` 无效 | 视为空白名单,不允许任何 MCP 服务器 |
| `allowManagedMcpServersOnly` 无效 | 视为 `true` |
| `requiredMinimumVersion` / `requiredMaximumVersion` 无效 | 剥离而非强制(避免错误策略阻止启动) |

> **类比**:像"政策文件中一个错别字不会废掉整个政策"——无效部分被丢弃,有效部分继续执行,安全相关字段则 fail-safe(宁可更严格)。

验证方式:运行 `claude doctor` 查看每个无效条目及其来源。
