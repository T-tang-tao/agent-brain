## 工具分类

Hermes 内置 70+ 工具,分布在约 28 个工具集中。按能力域分组如下:

| 分类 | 示例工具 | 说明 |
|---|---|---|
| Web | `web_search`, `web_extract` | 网页搜索和内容提取 |
| X Search | `x_search` | 搜索 X(Twitter)帖子,需 xAI 凭证 |
| 终端与文件 | `terminal`, `process`, `read_file`, `patch` | 执行命令和操作文件 |
| 浏览器 | `browser_navigate`, `browser_snapshot`, `browser_vision` | 交互式浏览器自动化 |
| 媒体 | `vision_analyze`, `image_generate`, `text_to_speech` | 多模态分析和生成 |
| Agent 编排 | `todo`, `clarify`, `execute_code`, `delegate_task` | 规划、澄清、代码执行、子 Agent 委托 |
| 记忆与回忆 | `memory`, `session_search` | 持久记忆和会话搜索 |
| 自动化 | `cronjob` | 定时任务管理 |
| 集成 | `ha_*`, MCP server tools | Home Assistant、MCP 等 |

> **类比**:工具集像"应用商店的分类标签"——你可以按需启用整个分类,而不是逐个工具挑选。

## 终端后端

`terminal` 工具支持六种执行后端,适配从本地开发到云端 serverless 的各种场景:

| 后端 | 说明 | 适用场景 |
|---|---|---|
| `local` | 本地直接执行 | 日常开发、快速迭代 |
| `docker` | Docker 容器隔离 | 需要环境隔离的任务 |
| `ssh` | 远程 SSH 主机 | 远程服务器操作 |
| `singularity` | Singularity 容器(HPC) | 高性能计算集群 |
| `modal` | Modal serverless | 云端按需计算,空闲休眠 |
| `daytona` | Daytona workspace | 云端持久化工作区,按需唤醒 |

后两者支持 **serverless 持久化**——空闲时休眠,按需唤醒,兼顾成本与状态保留。

## 后台进程管理

长耗时任务可后台运行,通过 `process` 工具管理生命周期:

```python
# 启动后台进程,立即返回 session_id 和 pid
terminal(command="pytest -v tests/", background=true)

# 列出所有运行中进程
process(action="list")

# 检查特定进程状态
process(action="poll", session_id="...")

# 终止进程
process(action="kill", session_id="...")

# 向交互式进程发送输入
process(action="write", session_id="...", data="y")
```

### PTY 模式

`terminal` 支持 `pty=true` 参数,启用伪终端模式。PTY 模式用于交互式 CLI 工具(如 Codex、Claude Code),这类工具需要完整的终端行为(光标控制、颜色渲染、行缓冲)。

```python
terminal(command="claude", pty=true)
```

## Sudo 支持

命令需要 sudo 权限时:

1. **交互提示**:Hermes 会提示用户输入密码,密码在会话内缓存,后续 sudo 命令无需重复输入。
2. **预配置**:在 `~/.hermes/.env` 设置 `SUDO_PASSWORD`,实现无人值守场景下的自动提权。

```bash
# ~/.hermes/.env
SUDO_PASSWORD=your_password_here
```

## 容器安全

所有容器后端(Docker、Singularity)均经过安全加固,确保 Agent 执行的命令不会逃逸到宿主系统:

| 加固项 | 说明 |
|---|---|
| 只读根文件系统 | Docker 根文件系统挂载为只读 |
| 丢弃所有 capabilities | 所有 Linux capabilities 被丢弃 |
| 禁止权限提升 | `no-new-privileges` 标志 |
| PID 限制 | 最多 256 进程,防止 fork 炸弹 |
| 完整命名空间隔离 | PID、网络、挂载、用户等命名空间全隔离 |

## 足迹阶梯

工具扩展遵循 [01-概述](./01-概述.md) 中的足迹阶梯原则——优先选择足迹小的扩展方式,新增核心工具是最后手段:

```
足迹从小到大:
1. 扩展现有代码          ← 零新增表面
2. CLI 命令 + Skill      ← 零模型工具足迹(默认选择)
3. 服务门控工具(check_fn) ← 配置后才出现
4. Plugin                ← 第三方/小众/用户特定
5. MCP Server(目录中)    ← 零核心 schema 足迹,任何 MCP host 可复用
6. 新核心工具            ← 最后手段,仅限基础且通用能力
```

**策略含义**:

| 阶梯 | 模型工具足迹 | 适用场景 |
|---|---|---|
| 扩展现有代码 | 零 | 能力增强,不引入新接口 |
| CLI + Skill | 零 | 默认选择,大多数新能力走这条路 |
| 服务门控工具 | 配置后才出现 | 需要 API key 或特定环境 |
| Plugin | 第三方注册 | 小众、用户特定能力 |
| MCP Server | 零 schema | 通用能力,跨 host 复用 |
| 新核心工具 | 永久增加 | 仅限基础且通用,门槛最高 |

> **核心约束**:每个模型工具都会随每次 API 调用发送,新增核心工具直接增加每轮 token 成本。这就是"窄腰宽边"原则——核心保持精简,能力在边缘扩展。
