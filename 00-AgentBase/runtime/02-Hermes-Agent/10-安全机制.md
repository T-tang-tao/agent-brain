## 七层安全模型

Hermes 采用纵深防御（defense-in-depth）安全模型——没有单一防线足以应对所有威胁，多层叠加使攻击者必须同时突破每一层才能造成实质损害。

| 层 | 说明 |
|---|---|
| 1. 用户授权 | 谁能和 Agent 对话（允许列表、DM 配对） |
| 2. 危险命令审批 | 破坏性操作的人工确认 |
| 3. 容器隔离 | Docker/Singularity/Modal 沙箱加固 |
| 4. MCP 凭证过滤 | MCP 子进程的环境变量隔离 |
| 5. 上下文文件扫描 | 项目文件中的 prompt injection 检测 |
| 6. 跨会话隔离 | 会话间不能互相访问数据；cron 存储路径防遍历攻击 |
| 7. 输入净化 | 终端工具后端的工作目录参数验证 |

> 类比：一座城堡——外层护城河（用户授权）、城门卫兵（命令审批）、内城墙（容器隔离）、金库（凭证过滤），层层设防。

## 危险命令审批

### 审批模式

```yaml
approvals:
  mode: manual    # manual | smart | off
  timeout: 60     # 等待用户响应秒数
  cron_mode: deny # deny | approve（cron 无头模式行为）
```

| 模式 | 行为 |
|---|---|
| manual（默认） | 总是提示用户审批危险命令 |
| smart | 用辅助 LLM 评估风险，低风险自动批准，危险自动拒绝，不确定时升级为手动提示 |
| off | 禁用所有审批检查 |

### YOLO 模式

三种激活方式：

- CLI 标志：`hermes --yolo`
- 斜杠命令：`/yolo`
- 环境变量：`HERMES_YOLO_MODE=1`

YOLO 是**切换**——每次使用翻转开关（开→关→开）。

激活时显示两个持续视觉提醒：

1. 会话开始红色横幅
2. 状态栏 ⚠ YOLO 标记

### 硬线黑名单（始终开启的底线）

某些命令如此灾难性，Hermes 无论如何都拒绝执行——即使 YOLO 开启、`approvals.mode: off`：

- `rm -rf /` 及变体
- bash fork bomb
- `mkfs` 在挂载的根设备
- `dd if=/dev/zero of=/dev/sd*`
- 管道不受信任 URL 到 `sh`

> 硬线黑名单不可被任何配置绕过，是安全的最后一道不可移动的底线。

### 触发审批的模式

| 模式类别 | 示例 |
|---|---|
| 递归删除 | `rm -r` |
| 危险权限 | `chmod 777` |
| 文件系统格式化 | `mkfs` |
| 磁盘直写 | `dd` |
| 数据库破坏 | `DROP TABLE`、`DELETE FROM`（无 WHERE） |
| 系统配置覆写 | `> /etc/` |
| 服务停止 | `systemctl stop` |
| 批量终止 | `kill -9 -1` |
| 远程执行 | `curl|sh` |
| 配置就地编辑 | `sed -i /etc/` |

**容器内跳过检查**：Docker/Singularity/Modal/Daytona 后端中，容器本身就是安全边界，故跳过审批检查。

### 审批流程

**CLI 中四选项**：

| 选项 | 含义 |
|---|---|
| once | 单次批准 |
| session | 本会话有效 |
| always | 永久允许（写入配置） |
| deny | 拒绝（默认） |

**Gateway 中**：回复 `yes`/`approve` 或 `no`/`deny`。

### 永久允许列表

"always" 批准的命令保存到 `config.yaml` 的 `command_allowlist`，后续遇到相同命令模式时自动放行。

## 用户授权（网关）

授权检查顺序（逐层短路，命中即返回）：

1. Per-platform allow-all 标志
2. DM 配对已批准列表
3. 平台特定允许列表
4. 全局允许列表
5. 全局 allow-all
6. 默认：**拒绝**

> 未配置任何允许列表时，所有用户被拒绝——安全默认是"不开门"。

## DM 配对系统

配对码特性：

| 特性 | 值 |
|---|---|
| 长度 | 8 字符 |
| 字母表 | 32 字符无歧义字母表 |
| 随机性 | 加密随机 |
| 过期时间 | 1 小时 |
| 限速 | 10 分钟内限制生成频率 |
| 待处理上限 | 最多 3 个待处理码 |
| 锁定策略 | 5 次失败尝试后锁定 1 小时 |
| 文件权限 | `chmod 0600` |

## 容器隔离

Docker 安全标志：

```bash
--cap-drop ALL                          # 丢弃所有 Linux capabilities
--cap-add DAC_OVERRIDE                  # Root 可写入绑定挂载目录
--cap-add CHOWN                         # 包管理器需要文件所有权
--cap-add FOWNER                        # 包管理器需要文件所有权
--security-opt no-new-privileges        # 阻止权限提升
--pids-limit 256                        # 限制进程数
--tmpfs /tmp:rw,nosuid,size=512m        # 限制 /tmp 大小
--tmpfs /var/tmp:rw,noexec,nosuid       # /var/tmp 不可执行
--tmpfs /run:rw,noexec,nosuid           # /run 不可执行
```

资源限制配置项：

| 配置 | 说明 |
|---|---|
| `container_cpu` | CPU 配额 |
| `container_memory` | 内存上限 |
| `container_disk` | 磁盘空间 |
| `container_persistent` | 持久化存储 |

> 容器隔离是第三层防线：即使命令审批被绕过（如 YOLO 模式），破坏性操作也被限制在一次性容器内，无法触及宿主系统。
