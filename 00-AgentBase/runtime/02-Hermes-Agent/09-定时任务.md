## 这是什么

Hermes 内置 **cron 调度器**，可用自然语言或 cron 表达式调度任务。通过单一 `cronjob` 工具管理，支持创建、暂停、恢复、编辑、触发、删除。

类比来说：它就像一个"AI 秘书的闹钟"。你告诉它"每天早上 9 点查一下新闻"，它就会按时唤起 Agent 执行任务，再把结果送到你手上。

## 创建定时任务

三种方式：

```bash
# 聊天中
/cron add 30m "Remind me to check the build"
/cron add "every 2h" "Check server status"

# CLI
hermes cron create "every 2h" "Check server status"

# 自然语言
Every morning at 9am, check Hacker News for AI news and send me a summary on Telegram.
```

| 方式 | 场景 |
|---|---|
| 聊天 `/cron add` | 在对话中快速创建 |
| CLI `hermes cron create` | 脚本化批量创建 |
| 自然语言 | 直接描述意图，Hermes 自动解析 |

## 技能绑定

Cron 任务可加载一个或多个技能：

```bash
/cron add "every 1h" "Summarize new feed items" --skill blogwatcher
/cron add "every 1h" "Combine results" --skill blogwatcher --skill maps
```

加载技能后，任务执行时该技能的行为定义会被注入 Agent 上下文，等同于临时启用对应能力。

## 工作目录

默认不加载任何项目上下文。通过 `--workdir` 指定项目目录：

```bash
hermes cron create "every 1d at 09:00" "Audit open PRs" --workdir /home/me/projects/acme
```

设置后 `AGENTS.md`、`CLAUDE.md` 等上下文文件被注入，工具使用该目录作为工作目录。

## 编辑任务

无需删除重建，直接编辑：

```bash
/cron edit <job_id> --schedule "every 4h"
/cron edit <job_id> --prompt "Revised task"
/cron edit <job_id> --skill blogwatcher --skill maps
/cron edit <job_id> --remove-skill blogwatcher
```

## 生命周期操作

```bash
/cron list                # 列出所有任务
/cron pause <job_id>      # 暂停
/cron resume <job_id>     # 恢复
/cron run <job_id>        # 立即触发
/cron remove <job_id>     # 删除
```

`job_id` 也接受任务名称（不区分大小写）。

## 调度器行为

Cron 执行由 gateway 守护进程处理。网关每 60 秒 tick 一次：

1. 从 `~/.hermes/cron/jobs.json` 加载任务
2. 检查 `next_run_at`
3. 为每个到期任务启动新 AIAgent 会话
4. 可选注入技能
5. 运行 prompt 到完成
6. 投递响应
7. 更新运行元数据和下次调度时间

文件锁防止重叠 tick 重复运行。

```
┌──────────────────────────────────────────────┐
│              Gateway 每 60s tick              │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │ 1. 加载 jobs.json                       │  │
│  │ 2. 检查 next_run_at                     │  │
│  │ 3. 到期任务 → 新 AIAgent 会话            │  │
│  │ 4. 注入技能（可选）                      │  │
│  │ 5. 运行 prompt                          │  │
│  │ 6. 投递响应                              │  │
│  │ 7. 更新元数据 & next_run_at             │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  文件锁: 防止重叠 tick 重复运行               │
└──────────────────────────────────────────────┘
```

## 投递选项

| 选项 | 说明 |
|---|---|
| origin | 投递到创建任务的位置（消息平台默认） |
| local | 仅保存到本地文件（CLI 默认） |
| telegram / discord / slack / ... | 投递到特定平台 |
| all | 扇出到所有已连接的 home channel |
| telegram,discord | 逗号分隔多平台 |

## 响应包装

默认投递输出带标题和页脚。设置 `cron.wrap_response: false` 投递原始输出。

```yaml
cron:
  wrap_response: false   # 投递原始输出，不带包装
```

## 可继续任务

设置 `cron.mirror_delivery: true` 使投递可回复——Agent 在上下文中有完整简报，用户可直接在投递消息后追问。

```yaml
cron:
  mirror_delivery: true
```

## No-Agent 模式

Cron 任务可运行在 **no-agent 模式**——按计划执行脚本，stdout 原样投递，零 LLM 参与。

适用于纯脚本任务（如定时备份、日志清理），无需 LLM 推理，节省 token 成本。

## Provider 快照

创建时未指定 provider/model 的任务会**快照当前全局默认**。如果全局默认后来改变，任务 fail closed——跳过运行并发送告警。

| 情况 | 行为 |
|---|---|
| 创建时指定 provider/model | 固定使用该配置 |
| 创建时未指定 | 快照当时全局默认 |
| 全局默认变更后 | 未指定任务 fail closed + 告警 |

## 安全限制

Cron 运行的会话**不能递归创建更多 cron 任务**，防止调度循环。
