## 记忆机制

两个文件构成 Agent 的记忆，让 Hermes 从"临时工"变成"长期合伙人"——记住你、越用越懂你：

| 文件 | 用途 | 字符限制 |
|---|---|---|
| MEMORY.md | Agent 个人笔记——环境事实、约定、学到的东西 | 2,200 字符 (~800 token) |
| USER.md | 用户画像——偏好、沟通风格、期望 | 1,375 字符 (~500 token) |

文件存储在 `~/.hermes/memories/`，会话开始时作为**冻结快照**注入系统提示。

> 类比：MEMORY.md 像 Agent 的"工作笔记本"，USER.md 像"客户档案"——前者记环境与约定，后者记你是谁。

## 冻结快照模式

系统提示注入在会话开始时捕获一次，会话中不变。这是刻意的——保持 LLM 的 prefix 缓存以提升性能。

Agent 在会话中添加/删除记忆条目时，变更**立即持久化到磁盘**，但不会在系统提示中体现，直到下一个会话。

> 类比：像 Git 的 commit——一旦写入会话就不可变，修改要等下一次 commit（新会话）才生效。这与 Hermes "对话级 Prompt 缓存是神圣的"核心原则一致。

## 记忆工具操作

Agent 使用 `memory` 工具管理记忆：

| 操作 | 说明 |
|---|---|
| `add` | 添加新记忆条目 |
| `replace` | 用子串匹配替换现有条目 |
| `remove` | 移除不再相关的条目 |

没有 `read` 操作——记忆内容在会话开始时自动注入，无需主动读取。

## 什么该存 vs 什么不该存

记忆容量有限，必须精打细算：

| 该存 | 不该存 |
|---|---|
| 用户偏好 | 琐碎信息 |
| 环境事实 | 易重新发现的事实 |
| 纠正 | 原始数据转储 |
| 约定 | 会话特定临时信息 |
| 已完成的工作 | 已在上下文文件中的信息 |
| 明确请求 | |

## 容量管理

| 存储 | 限制 | 典型条目数 |
|---|---|---|
| memory | 2,200 字符 | 8-15 条 |
| user | 1,375 字符 | 5-10 条 |

满时返回错误，Agent 需自行合并或删除条目后重试。

**最佳实践**：达到 80% 容量时开始合并——把零散条目归纳为更紧凑的表述，腾出空间给新记忆。

## 安全扫描

记忆条目在写入前扫描注入和泄露模式。匹配以下威胁模式或包含不可见 Unicode 字符的内容被阻止：

- prompt injection（提示注入）
- 凭证泄露
- SSH 后门
- 不可见 Unicode 字符

> 这是 Hermes 七层安全模型的一环——防止恶意内容通过记忆系统持久化并污染后续会话。

## 会话搜索

除 MEMORY.md 和 USER.md 外，Agent 可用 `session_search` 搜索过去对话：

- 所有 CLI 和消息会话存储在 SQLite（`~/.hermes/state.db`），带 FTS5 全文搜索
- 搜索查询返回**实际消息**——无 LLM 摘要、无截断
- 可在找到的会话中前后滚动，查看完整上下文

### 持久记忆 vs 会话搜索

| 特性 | 持久记忆 | 会话搜索 |
|---|---|---|
| 容量 | ~1,300 token | 无限（所有会话） |
| 速度 | 即时（在系统提示中） | ~20ms FTS5 查询 |
| 成本 | 每次提示的 token 成本 | 免费——无 LLM 调用 |
| 用途 | 始终需要的关键事实 | 查找特定过去对话 |

> 类比：持久记忆像"随身笔记本"（小而精，随时可看），会话搜索像"档案馆"（大而全，按需检索）。

## write_approval 写入审批

默认 Agent 自由保存记忆。设置 `memory.write_approval: true` 后，所有写入需要审批：

| 场景 | 行为 |
|---|---|
| CLI 中前台写入 | 内联提示，即时审批 |
| 消息平台 / 后台审查 | 写入暂存，用 `/memory pending` 审查 |

## 后台自我改进审查

每轮对话后，后台审查可能安静地保存记忆或更新技能。

| 配置 | 作用 |
|---|---|
| `display.memory_notifications` | 控制通知级别：`off` / `on` / `verbose` |
| `auxiliary.background_review` | 配置审查的 provider 和 model |

可在更便宜的模型上运行审查，降低成本——主模型专注对话，辅助模型负责自我改进。

## 外部记忆 Provider

8 个外部记忆 provider 插件，与内置记忆**并行运行**（不替换），添加知识图谱、语义搜索、自动事实提取等能力：

- Honcho
- OpenViking
- Mem0
- Hindsight
- Holographic
- RetainDB
- ByteRover
- Supermemory

```bash
hermes memory setup   # 选择并配置 provider
hermes memory status  # 检查活跃状态
```

## 配置

```yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
  write_approval: false
```
