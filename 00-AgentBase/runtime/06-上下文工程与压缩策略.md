# 上下文工程与压缩策略

## 一句话解释

上下文工程是管理 Agent 看到什么、保留什么、遗忘什么的工程能力。

Agent 的质量不只取决于模型,还取决于上下文是否准确、足够、干净、可追溯。

## 上下文的组成

| 类型 | 内容 | 风险 |
|---|---|---|
| 系统指令 | 角色、边界、输出要求 | 太长会淹没任务 |
| 项目规则 | AGENTS.md、README、约定 | 过期会误导 |
| 当前任务 | 用户目标、最新补充 | 被旧任务污染 |
| 工作记忆 | 计划、已做操作、工具结果 | 越积越乱 |
| 检索材料 | 文档、代码片段、搜索结果 | 片段不完整 |
| 长期记忆 | 用户偏好、稳定经验 | 错误记忆会放大 |
| 外部状态 | 文件、数据库、网页、工单 | 可能已变化 |

## 上下文优先级

当上下文冲突时,按以下顺序处理:

1. 当前用户最新明确要求。
2. 当前项目根级和子目录 `AGENTS.md`。
3. 系统和运行时安全规则。
4. 目标文件和相邻实现。
5. 历史对话和长期记忆。
6. 推断和经验。

推断永远不能覆盖明确事实。

## 上下文收集流程

1. 先读入口规则:项目 `AGENTS.md`、README、目录结构。
2. 再读任务相关文件:入口、调用链、测试、配置。
3. 只检索和任务相关的内容,避免把整个仓库塞进上下文。
4. 把发现转成短事实,不要保留大段原文。
5. 对关键事实记录来源路径或工具结果。

## 压缩策略

| 策略 | 做法 | 适用场景 |
|---|---|---|
| 摘要压缩 | 把长文档压缩成事实列表 | 长 README、规范文档 |
| 结构压缩 | 保留目录树、模块边界、调用链 | 大型代码库 |
| 证据压缩 | 保留文件路径、函数名、关键行 | 代码修改和评审 |
| 状态压缩 | 保留已完成、待做、阻塞、风险 | 长任务 |
| 决策压缩 | 保留为什么选择某方案 | 设计任务 |
| 失败压缩 | 保留失败命令、报错、已排除原因 | 调试任务 |

不要把压缩做成“泛泛总结”。好的压缩应能支持下一步行动。

## 上下文窗口不够时

优先保留:

1. 用户最新目标和限制。
2. 当前计划和未完成事项。
3. 直接相关文件的结构和关键片段。
4. 已经运行过的验证结果。
5. 不能重复或不能破坏的操作记录。

可以丢弃:

- 已经完成且不影响后续的探索过程。
- 无关候选方案。
- 大段重复日志。
- 已被更准确信息替代的旧假设。

## RAG 与直接阅读

| 场景 | 优先方式 |
|---|---|
| 知道具体文件 | 直接读文件 |
| 不知道位置 | 搜索 / RAG |
| 需要精确修改代码 | 直接读完整相关片段 |
| 需要了解大范围背景 | 先 RAG,再读原文 |
| 需要引用来源 | 保留路径和行号 |

RAG 解决“找得到”,不保证“读完整”。涉及实现、权限、SQL、接口契约时,必须回到原文件确认。

## 长任务的上下文协议

长任务中 Agent 应持续维护:

- 当前目标
- 已确认事实
- 当前计划
- 已完成操作
- 待验证事项
- 阻塞点
- 风险和回滚点

当上下文即将压缩或任务被恢复时,这六项比完整历史更重要。

## 常见失败模式

| 失败模式 | 表现 | 防线 |
|---|---|---|
| 上下文过宽 | 读太多无关内容,判断发散 | 先定模块和问题边界 |
| 上下文过窄 | 漏掉调用链或配置 | 读相邻实现和测试 |
| 旧信息污染 | 按过期事实行动 | 以最新用户消息和文件为准 |
| 片段幻觉 | 根据片段猜完整逻辑 | 回到原文件确认 |
| 总结失真 | 压缩后丢失限制条件 | 保留约束和来源 |
