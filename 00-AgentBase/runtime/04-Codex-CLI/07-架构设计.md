## 技术栈

| 方面 | 选择 | 原因 |
|------|------|------|
| 语言 | Rust | 性能、安全、跨平台 |
| TUI 框架 | 自研（基于 crossterm） | 终端交互控制 |
| 配置格式 | TOML | 人类可读，Rust 生态原生支持 |
| 沙箱 | OS 原生（Seatbelt/bwrap/seccomp） | 最小权限隔离 |
| 序列化 | serde + serde_json | Rust 生态标准 |

## Rust 工作区结构

```
codex/
├── Cargo.toml                 # 工作区根配置
├── codex-rs/                  # Rust 核心实现
│   ├── core/                  # 核心库：配置、模型调用、工具执行
│   ├── tui/                   # 终端 UI
│   ├── exec/                  # 命令执行引擎
│   ├── mcp-server/            # MCP 服务器实现
│   └── protocol/              # 内部协议定义
├── codex-js/                  # Node.js 绑定（npm 分发）
├── docs/                      # 文档
└── scripts/                   # 构建/安装脚本
```

| 模块 | 职责 |
|------|------|
| `core/` | 配置加载、模型 API 调用、工具注册、沙箱管理 |
| `tui/` | 终端界面渲染、用户交互、输入处理 |
| `exec/` | 命令执行、沙箱封装、进程管理 |
| `mcp-server/` | MCP 协议实现（Host 端） |
| `protocol/` | 内部消息类型和协议定义 |
| `codex-js/` | Node.js 包装层，用于 npm 分发 |

## 核心执行循环

```
┌──────────────────────────────────────────────────────┐
│                    Codex 执行循环                      │
│                                                      │
│  ┌─────────┐    ┌──────────┐    ┌───────────────┐   │
│  │ 用户输入 │───→│ 模型调用  │───→│ 工具调用决策   │   │
│  └─────────┘    └──────────┘    └───────┬───────┘   │
│                        ↑                  │          │
│                        │           ┌──────▼──────┐   │
│                        │           │ 沙箱检查     │   │
│                        │           └──────┬──────┘   │
│                        │           ┌──────▼──────┐   │
│                        │           │ 审批检查     │   │
│                        │           └──────┬──────┘   │
│                        │           ┌──────▼──────┐   │
│                        │           │ 执行工具     │   │
│                        │           └──────┬──────┘   │
│                        │           ┌──────▼──────┐   │
│                        └───────────│ 结果反馈     │   │
│                                    └─────────────┘   │
└──────────────────────────────────────────────────────┘
```

### 循环步骤

1. **用户输入**：接收用户指令或任务
2. **模型调用**：将上下文发送给 LLM，获取响应
3. **工具调用决策**：模型决定是否需要调用工具
4. **沙箱检查**：验证操作是否在沙箱允许范围内
5. **审批检查**：根据审批策略决定是否需要用户确认
6. **执行工具**：在沙箱中执行工具调用
7. **结果反馈**：将执行结果返回给模型，进入下一轮

> **关键**：沙箱检查和审批检查是两个独立的关卡。沙箱不过，操作直接拒绝；沙箱过了但审批需要确认，暂停等待用户。

### TUI 特性

| 特性 | 说明 |
|------|------|
| 流式输出 | 模型响应实时流式显示 |
| Markdown 渲染 | 代码块语法高亮、表格渲染 |
| 工具输出实时显示 | 工具执行过程可见 |
| 中断支持 | 可中途取消模型调用和工具执行 |
| 历史导航 | 上下键浏览对话历史 |
| 多行输入 | 支持多行编辑 |

## 配置生命周期

```
启动时:
  1. 读取 CODEX_HOME 环境变量（默认 ~/.codex/）
  2. 加载系统配置 (/etc/codex/config.toml)
  3. 加载用户配置 (~/.codex/config.toml)
  4. 加载 Profile 配置 (--profile 指定)
  5. 加载项目配置 (.codex/config.toml)
  6. 应用 CLI 标志覆盖
  7. 合并所有配置层，生成最终配置

运行时:
  8. 根据 sandbox_mode 初始化沙箱
  9. 根据 approval_policy 设置审批回调
  10. 注册内置工具
  11. 连接 MCP 服务器
  12. 加载 AGENTS.md 上下文
  13. 加载 Skills 元数据（Level 0）
  14. 进入执行循环
```

## 工具注册与调用

```
工具来源:
  ├── 内置工具 (shell, file_read, file_write, web_search...)
  ├── MCP 工具 (STDIO/HTTP 服务器提供)
  └── 插件工具 (插件注册)

调用流程:
  模型输出 tool_call
       ↓
  工具注册表查找
       ↓
  沙箱权限检查 → 拒绝 → 返回错误
       ↓ 通过
  审批策略检查 → 需确认 → 等待用户
       ↓ 通过
  执行工具
       ↓
  返回结果给模型
```

## 与 Hermes / Claude Code 的架构对比

| 方面 | Codex CLI | Hermes Agent | Claude Code |
|------|-----------|--------------|-------------|
| 语言 | Rust | Python + TypeScript | TypeScript |
| 沙箱 | OS 原生（Seatbelt/bwrap） | 容器（Docker/Singularity） | OS 原生 + 容器 |
| 配置 | TOML | YAML + env | JSON |
| TUI | 自研 | Ink (React) | Ink (React) |
| 多平台 | 终端 only | 终端 + 20+ 消息平台 | 终端 + IDE + Web + 桌面 |
| 执行后端 | 本地沙箱 | 6 种（local/docker/ssh/modal/daytona/singularity） | 本地 + Docker |
| 记忆系统 | 实验性（memories feature） | 内置（SQLite + FTS5 + Honcho） | 内置（CLAUDE.md） |
| 自我进化 | 无 | 有（自动创建技能） | 无 |

> **Codex 的独特定位**：最轻量、最安全、最 Rust-native。适合本地开发场景，不追求多平台覆盖，而是把"在终端安全地编码"这件事做到极致。

## 设计原则

| 原则 | 体现 |
|------|------|
| **安全优先** | 两层防御（沙箱+审批），默认最安全配置 |
| **最小权限** | 默认阻断网络，白名单放行；shell 环境默认仅继承核心变量 |
| **配置分层** | 6 级优先级，灵活组合 |
| **渐进式扩展** | AGENTS.md → Skills → MCP → Subagents，按需升级 |
| **跨平台一致** | 不同 OS 用不同沙箱技术，但行为一致 |
