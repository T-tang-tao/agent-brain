## 两层安全模型

Codex 的安全设计由两个正交层组成：

```
┌───────────────────────────────────────────────────┐
│  沙箱层（Sandbox）—— 控制能力边界                    │
│  决定 Agent 能读写什么、能否联网                      │
│                                                   │
│  ┌─────────────────────────────────────────────┐  │
│  │  审批层（Approval）—— 控制确认机制            │  │
│  │  决定哪些操作需要人工批准                      │  │
│  └─────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────┘
```

| 层 | 解决的问题 | 类比 |
|----|----------|------|
| 沙箱 | Agent **能做什么** | 给员工配钥匙：能进哪些房间 |
| 审批 | Agent 做了之后**需不需要批准** | 重要操作需要经理签字 |

## 沙箱模式

三种沙箱模式，能力递增：

### read-only（只读）

```toml
sandbox_mode = "read-only"
```

- 文件系统：只能读，不能写
- 网络：默认阻断
- 适用：代码审查、只读分析、学习理解代码库

### workspace-write（工作区写入）

```toml
sandbox_mode = "workspace-write"
```

- 文件系统：工作区内可读写，工作区外只读
- 网络：默认阻断，可通过 network_proxy 放行
- 受保护路径：`.git`、`.agents`、`.codex` 目录只读
- 适用：日常开发（默认推荐）

### danger-full-access（完全访问）

```toml
sandbox_mode = "danger-full-access"
```

- 文件系统：无限制
- 网络：无限制
- 适用：可信环境下的自动化任务，**谨慎使用**

## 审批策略

四种审批策略，自动化程度递增：

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `untrusted` | 所有操作都需确认 | 不可信代码、审查陌生项目 |
| `on-request` | 写操作和命令需确认（默认） | 日常开发 |
| `never` | 全自动，不确认 | 可信环境下的自动化 |
| `granular` | 细粒度，按工具配置 | 精确控制特定工具 |

```toml
# config.toml
approval_policy = "on-request"
```

```bash
# CLI 标志
codex --approval untrusted
codex --approval never
```

### 快捷模式

```bash
# 全自动：sandbox=workspace-write + approval=never
codex --full-auto
```

### auto_review 自动审查

```toml
approvals_reviewer = "auto_review"
```

启用后，Agent 在提交操作前会自动进行自我审查，判断操作是否安全，减少人工确认需求。

## 沙箱与审批的组合矩阵

| 沙箱 \ 审批 | untrusted | on-request | never | granular |
|-------------|-----------|------------|-------|----------|
| read-only | 最安全 | 安全 | 自动只读 | 精细只读 |
| workspace-write | 安全 | **默认推荐** | 全自动 | 精细开发 |
| danger-full-access | 谨慎 | 谨慎 | 最危险 | 精细全权 |

> **最佳实践**：日常用 `workspace-write` + `on-request`（默认），自动化用 `workspace-write` + `never`（`--full-auto`），审查陌生代码用 `read-only` + `untrusted`。

## 网络隔离

### 默认阻断

`workspace-write` 和 `read-only` 模式下，网络默认完全阻断。Agent 无法访问任何网络资源。

### network_proxy 域名白名单

```toml
[network_proxy]
allow = [
    "api.example.com",
    "registry.npmjs.org",
    "pypi.org"
]
```

只有白名单中的域名可以访问。其他所有网络请求被阻断。

### DNS 重绑定保护

Codex 防止 DNS 重绑定攻击：即使域名在白名单中，如果 DNS 解析结果指向内网地址（如 127.0.0.1、10.x.x.x、192.168.x.x），请求仍会被阻断。

## 受保护路径

即使在工作区写入模式下，以下路径始终只读：

| 路径 | 原因 |
|------|------|
| `.git/` | 防止篡改 Git 历史 |
| `.agents/` | 保护 Agent 技能和配置 |
| `.codex/` | 保护 Codex 项目配置 |
| 工作区外 | 防止意外修改系统文件 |

## 操作系统级沙箱

Codex 根据操作系统使用不同的沙箱技术：

### macOS：Seatbelt

使用 Apple 的 `sandbox-exec`（Seatbelt）实现：

- 基于 Scheme 的策略语言
- 文件系统路径限制
- 网络socket 限制
- 进程限制

### Linux：bwrap + seccomp

使用 Bubblewrap + seccomp 实现：

- `bwrap`：命名空间隔离（挂载、PID、网络）
- `seccomp`：系统调用过滤
- 常用于容器化应用（Flatpak 同款技术）

### Windows：原生沙箱

Windows 原生沙箱支持为实验性：

- 基于 Windows AppContainer 或 Windows Sandbox
- 功能有限，推荐使用 WSL2 获得完整沙箱

## Dev Containers 支持

Codex 支持在 Dev Containers 中运行，提供更强的隔离：

```json
// .devcontainer/devcontainer.json
{
    "image": "mcr.microsoft.com/devcontainers/typescript-node",
    "features": {
        "ghcr.io/devcontainers/features/rust:1": {}
    }
}
```

在 Dev Container 中运行 Codex：

```bash
# 在 Dev Container 终端中
codex
```

优势：

- 完全隔离的文件系统
- 预装开发依赖
- 可重建环境
- 不影响宿主系统

## OTel 遥测

Codex 支持通过 OpenTelemetry (OTel) 发送遥测数据：

```toml
[otel]
endpoint = "http://localhost:4317"
service_name = "codex"
```

可采集的指标包括：

- 工具调用次数和耗时
- 模型调用次数和 token 消耗
- 沙箱违规事件
- 审批请求事件

> **用途**：生产环境监控 Agent 行为，审计安全事件。

## 安全检查清单

部署 Codex CLI 前的检查项：

- [ ] 选择合适的沙箱模式（默认 `workspace-write`）
- [ ] 选择合适的审批策略（默认 `on-request`）
- [ ] 配置 `shell_environment_policy`，排除敏感变量
- [ ] 检查 `network_proxy` 白名单，仅放行必要域名
- [ ] 确认受保护路径未被覆盖
- [ ] 生产环境考虑启用 OTel 遥测
- [ ] 如需更强隔离，使用 Dev Containers
