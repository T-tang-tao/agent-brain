# 如何构建 LLM Wiki

## 一句话解释

LLM Wiki 是一种同时给人和 Agent 读取的知识库。它不是简单堆 Markdown，也不是只做向量库，而是把原始资料、清洗后的知识、结构化架构图、可执行资产连接起来，让 LLM/Agent 能稳定理解、检索和使用。

本文件基于 `llm-wiki-skill` 这类项目体现的思想做基础知识整理：把零散资料转成可读、可维护、可迁移、可被 Agent 使用的 wiki。

## LLM Wiki 解决什么问题

普通笔记库主要服务人类阅读。  
普通 RAG 主要服务模型检索。  
LLM Wiki 试图同时服务两者：

1. 人能打开阅读，理解上下文；
2. Agent 能定位主题，抽取规则，执行任务；
3. 知识能持续更新，而不是一次性文档；
4. 不同 Agent 环境迁移时，仍能复用这套知识。

可以理解为：

```text
LLM Wiki = 人类可读的 wiki + Agent 可解析的结构 + 可迁移的知识资产
```

## 为什么不是只用向量数据库

向量数据库适合“相似内容检索”，但它不擅长表达：

- 知识层级；
- 文件之间的关系；
- 概念边界；
- 操作流程；
- 权限规则；
- 哪些内容是基础知识，哪些是可执行资产；
- 哪些信息已经过整理，哪些仍是原始材料。

所以 LLM Wiki 不应该只依赖向量库。

更合理的做法是：

```text
Markdown Wiki + 目录结构 + 双链 + 元数据 + 检索 + 模板 + 可执行资产
```

## 三层结构思想

构建 LLM Wiki 时，可以采用三层结构。

### 1. 原始层

原始层保存未经充分整理的资料。

例如：

- 原始文章；
- 项目 README；
- 对话记录；
- issue / PR 摘要；
- 外部文档摘录；
- 实验记录；
- 临时研究材料。

原始层的作用是保留来源，不要求马上变成最终知识。

### 2. 编译层

编译层把原始资料整理成更稳定的知识。

例如：

- 去重；
- 归纳概念；
- 拆分主题；
- 提炼定义；
- 整理对比表；
- 标记边界；
- 补充例子；
- 建立链接。

编译层是 LLM Wiki 的核心。它把“资料”变成“知识”。

### 3. 架构层

架构层提供全局地图和导航。

例如：

- 知识地图；
- 目录索引；
- 概念关系图；
- 模块边界；
- 推荐阅读顺序；
- 资产区与知识区对应关系。

架构层让人和 Agent 知道：

```text
我在哪里？这个主题和哪些文件相关？下一步该读什么？
```

## 当前 agent wiki 如何对应三层结构

当前 `agent/` 目录可以这样理解：

| 层级 | 当前对应 | 说明 |
|---|---|---|
| 原始层 | 后续可建立 `raw/` 或放入各迁移记录 | 保存未整理资料、外部摘录、迁移材料 |
| 编译层 | `00-AgentBase/` | 把 Agent、MCP、Skill、Prompt、Knowledge Base 等整理成基础知识 |
| 架构层 | `README.md`、`00-目录索引.md`、`08-Agent知识地图.md` | 提供入口、索引、总图和模块关系 |
| 资产层 | `01-*` 之后 | 放可直接使用的 Skill、Prompt、MCP、Boundary、Migration |

这里额外加了“资产层”，因为本目录不仅是学习 wiki，也是未来 agent 迁移资产库。

## 知识区和资产区要分开

LLM Wiki 最容易混乱的地方，是把解释性知识和可执行资产混在一起。

例如：

- “Skill 是什么”是知识；
- `debugging.skill.md` 是资产；
- “MCP 是什么”是知识；
- `github-mcp.md` 是资产；
- “Agent 为什么需要边界”是知识；
- `shell-command-policy.md` 是资产。

当前目录采用：

```text
00-AgentBase/  # 知识区
01-*           # 资产区
```

这能避免后续扩展时结构失控。

## 一个好的 LLM Wiki 文件应该长什么样

每篇知识文件建议包含：

```markdown
# 标题

## 一句话解释

## 这个文件解决什么问题

## 核心概念

## 详细说明

## 例子

## 常见误解

## 和其他概念的关系

## 相关文件
```

这对人友好，也对 Agent 友好。

Agent 读取时可以快速定位：

- 定义；
- 用途；
- 不是什么；
- 例子；
- 相关文件。

## LLM Wiki 的设计原则

### 1. 先有目录，再填内容

不要一开始就写很多散文档。

先定义：

- 哪些是知识区；
- 哪些是资产区；
- 哪些是原始资料；
- 哪些是迁移记录；
- 哪个文件是入口；
- 哪个文件是索引。

### 2. 每个文件只解决一个问题

例如：

- `01-AgentBase总览.md` 定义 Agent + 知识库入口；
- `02-概念全景辨析.md` 负责概念边界；
- `01-Agent知识库设计.md` 负责知识库设计。

不要把所有内容塞进一个大文件。

### 3. 用双链建立知识图谱

Obsidian 双链适合 LLM Wiki。

例如：

```markdown
- [[../01-AgentBase总览]]
- [[../02-概念全景辨析]]
- [[../03-知识地图]]
```

双链的作用不是好看，而是让人和 Agent 都能追踪上下文。

### 4. 写“边界”和“不是什么”

Agent 很容易混淆概念，所以知识文件应该写清楚：

- Skill 不是 MCP；
- MCP 不是知识库；
- Memory 不是 RAG；
- Plugin 不是 Prompt；
- Tool 不是 Agent。

“不是什么”有时比“是什么”更重要。

### 5. 保留来源，但不要照搬来源

LLM Wiki 应该能追溯来源，但最终知识要经过整理。

推荐流程：

```text
原始资料 → 摘录 → 清洗 → 归纳 → 写入 wiki → 链接相关文件 → 定期复查
```

### 6. 为 Agent 写可执行说明

如果某个文件未来可能被 Agent 使用，就要写清楚：

- 什么时候用；
- 输入是什么；
- 输出是什么；
- 哪些行为禁止；
- 相关知识在哪；
- 是否需要用户确认。

这尤其适用于 `01-*` 之后的资产区。

### 7. 区分静态知识和实时状态

静态知识可以写进 wiki。

例如：

- 概念解释；
- 操作流程；
- 项目约定；
- 迁移经验。

实时状态不应该写死。

例如：

- 当前价格；
- 当前账户余额；
- 当前订单状态；
- 当前服务健康状态；
- 最新 API 返回值。

实时状态应该通过工具、MCP 或 API 查询。

## LLM Wiki 的构建流程

可以按以下流程构建：

```text
1. 定义目标读者
2. 定义顶层目录
3. 区分知识区和资产区
4. 收集原始资料
5. 提炼核心概念
6. 拆成小文件
7. 建立索引和知识地图
8. 加双链
9. 建模板
10. 让 Agent 使用并反馈
11. 根据反馈持续修正
```

## 面向 Agent 的维护方式

未来 Agent 使用这个 wiki 时，应遵守：

1. 先读入口和索引；
2. 不确定概念时先查 `00-AgentBase/`；
3. 需要执行时再查 `01-*` 资产区；
4. 不把临时任务写进基础知识；
5. 新增资产时回链相关基础知识；
6. 发现过期知识时标记或修正；
7. 不把密钥、token、账户状态写入 wiki。

## 常见错误

### 错误 1：把所有资料都塞进一个 README

这样人难读，Agent 也难定位。

### 错误 2：把知识和模板混在一起

概念教程和可执行 prompt/skill/policy 应分开放。

### 错误 3：只做向量库，不做结构化目录

向量检索能找相似内容，但不能替代知识架构。

### 错误 4：没有索引和知识地图

没有索引时，wiki 很快会变成文件垃圾堆。

### 错误 5：不写边界

Agent 读到知识后可能会行动，所以必须知道哪些行为有限制。
