# 如何编写 AGENTS.md

> AGENTS.md 是给 Agent 看的管理文档,不是给人看的文章。规范决定了 Agent 行为的边界与可预测性。

## 一句话理解

AGENTS.md 是 **Agent 的管理规范**,定义 Agent 在该目录下能做什么、必须做什么、不能做什么。写法上**陈述事实**而不是**解释原因**。

## 与 README.md 的区别

| 维度 | README.md | AGENTS.md |
|------|-----------|-----------|
| 读者 | 人类 + Agent | Agent 为主 |
| 目的 | 介绍项目是什么 | 规定 Agent 怎么行为 |
| 风格 | 叙述、欢迎、营销 | 陈述、规则、约束 |
| 长度 | 灵活 | 简洁(避免冗余) |
| 更新频率 | 项目变化时 | 行为规范变化时 |

## AGENTS.md 的物理结构

每个项目 / 知识库 / 资产目录**可选**放一个 `AGENTS.md`,作为 Agent 进入该目录时的行为契约。

```
my-project/
├── AGENTS.md                ← Agent 进入时必读(本目录行为规范)
├── README.md                ← 人类入口(项目介绍)
├── src/
│   └── AGENTS.md            ← 子目录可单独定义(作用域更小)
└── tests/
    └── AGENTS.md            ← 测试目录专属行为规范
```

**作用域继承**:子目录 `AGENTS.md` 优先级高于父目录,父目录优先级高于根目录。

## 标准结构

| 章节 | 内容 | 长度建议 |
|------|------|----------|
| **核心理念** | 3-5 条铁律,每条 1 行 | ≤ 10 行 |
| **目录结构** | ASCII tree 注释当前结构 | 20-40 行 |
| **管理规范** | 入库/更新/删除/审计流程 | 按需 |
| **边界/禁区** | 哪些禁止做 | 按需 |
| **自检清单** | 操作前过一遍 | 按需 |

**不要**写的章节:
- ❌ "为什么这样设计"(→ 决策日志 99-Roadmap.md)
- ❌ "背景与起源"(→ 项目 README.md)
- ❌ "类似项目参考"(→ 知识库 references/)
- ❌ "完整原理说明"(→ 知识库 00-AgentBase/)

## 写作风格

### 1. 陈述事实,不解释原因

**反例** ❌

```markdown
**为什么不再分 `global/` 和 `projects/`?**

旧设计强制把技能分类到两个子目录。问题:

| 问题 | 后果 |
|------|------|
| 分类标准模糊 | "所有项目都会用到"vs"只在特定项目用到"经常判断错 |
| 一个技能可能多场景用 | 强制二选一不合适 |
| 分类是部署维度,不是技能属性 | 强制分类导致要重新组织才能改部署位置 |

**新原则**:技能平铺存放,部署位置由 AI 根据上下文判断...
```

**正例** ✅

```markdown
技能平铺存放,不分子目录。部署位置由 AI 根据上下文判断。
```

### 2. 规则用祈使句

**反例** ❌

```markdown
我们认为,也许可以考虑,在某些场景下,文件可能需要先备份...
```

**正例** ✅

```markdown
删除文件前必须先备份到 `.bak`。
```

### 3. 表格用结构化数据,不写散文

**反例** ❌

```markdown
状态有很多种,比如当一个东西刚被创建还没验证的时候就叫验证中,
当它通过验证可以正式用了就叫可用,还有就是当它被废弃了叫弃用...
```

**正例** ✅

```markdown
| 状态 | 含义 |
|------|------|
| 验证中 | 刚获取/生成,尚未验证可用性 |
| 可用 | 已通过验证,可投入编排 |
| 弃用 | 不再维护,待清理 |
```

### 4. 引用而不是重复

引用其它文件,而不是复制内容:

**反例** ❌

```markdown
[此处写 200 行解释 Skill 是什么]
```

**正例** ✅

```markdown
Skill 的定义见 [`../behavior/01-如何编写Skill.md`](../behavior/01-如何编写Skill.md)。
```

## 边界规则

AGENTS.md 应当明确以下边界,缺一不可:

- ✅ **允许做**:Agent 可自主执行的清单
- ⚠️ **需确认**:必须先问用户的清单
- ❌ **禁止做**:任何情况下禁止的清单

**示例** ✅

```markdown
## 边界

✅ 允许:读所有文件、写 01-Skills 实践层目录
⚠️ 需确认:写 00-AgentBase/、删除任何文件
❌ 禁止:覆盖 .obsidian/、把密钥写进任何 .md
```

## 入库与变更流程

| 操作 | 流程 |
|------|------|
| 新建 AGENTS.md | 复制根 AGENTS.md → 改 3 处:核心理念、目录结构、自检清单 |
| 修改 AGENTS.md | 直接改当前文件,不创建"AGENTS.md.bak" |
| 删除 AGENTS.md | 先确认父目录是否还需要 → 备份 → 删 |
| 跨目录统一 | 走 `01-Skills/agents-md-author/SKILL.md` 强制流程 |

## 与决策日志的关系

- **AGENTS.md**:写"现在是什么、必须怎么做"
- **99-Roadmap.md**:写"过去为什么这样决定、决定的历史"

**当用户说"不再分 global/projects/"时**:

- ❌ 不要在 AGENTS.md 里写"为什么不再分..."
- ✅ 在 AGENTS.md 里写"技能平铺存放"
- ✅ 在 99-Roadmap.md 决策日志里写"2026-07-09 — 决定平铺存放,理由:..."

## 反例:典型冗余段落

下面这些段落**都不应该出现在 AGENTS.md 里**:

```markdown
## 为什么这样设计?
## 设计动机
## 与其它方案的对比
## 取舍过程
## 历史背景
## 已知局限性
## 未来改进方向
```

这些段落要么放到 99-Roadmap.md(决策日志),要么放到知识库其它文档(原理说明)。

## 自检清单

写完 AGENTS.md 后,过一遍:

- [ ] 没有"为什么"段落(改放 99-Roadmap.md)
- [ ] 没有重复 00-AgentBase/ 里已有的内容
- [ ] 没有重复 README.md 里已有的内容
- [ ] 规则用祈使句,不是叙述句
- [ ] 表格用结构化数据,不是散文
- [ ] 边界规则覆盖了允许/确认/禁止三类
- [ ] 没有出现"我们认为""可能""也许"等模糊词
- [ ] 引用而不是重复其它文件内容
- [ ] 长度合理(单 AGENTS.md 通常 < 500 行)
