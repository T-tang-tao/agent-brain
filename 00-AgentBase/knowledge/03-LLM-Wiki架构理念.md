## 一句话理解

传统 RAG 是"外卖员":你提问时才去仓库现找现拼;LLM Wiki 是"农场主":在你提问之前就把食材种好、加工好、分类存好。

## 谁提出的

Andrej Karpathy(前 OpenAI 联合创始人、前特斯拉 AI 负责人)提出的一种 AI 知识库架构理念。核心是**把 LLM 的角色从"阅读工具"转变为"知识编译器"**。

## 核心问题:RAG 为什么不够

传统 RAG(检索增强生成)的流程是:用户提问 → 从原始文档库检索相关片段 → LLM 基于片段生成回答。问题在于:

- **原始文档没有经过加工**:散落、可能矛盾、包含大量无关信息
- **每次查询都是一次性的**:用完就忘,不随时间积累
- **矛盾信息无法处理**:两份说法冲突的文档都会被召回,LLM 不知道信哪个
- **检索噪声高**:原始文档的粒度不一定匹配问题

LLM Wiki 的解法:**把处理时机从"查询时"提前到"写入时"**。

## 三层架构

```
┌──────────────────────────────────────────────────┐
│  Raw Sources(原始资料层)                          │
│                                                    │
│  只读的原始来源:文档、代码、对话记录、外部摘录      │
│  不被修改,仅作为编译的"源码"                       │
│  保留溯源能力:每个知识点可追溯回原始材料            │
├──────────────────────────────────────────────────┤
│  Wiki(知识层)                                    │
│                                                    │
│  由 LLM 全权编写的结构化知识文件(Markdown)         │
│  三种产物:                                         │
│  · 基础页面:原子知识单元(一个概念、一个操作)      │
│  · 高阶页面:多对一聚合(概念全景、对比分析)        │
│  · 知识图谱:显式存储节点和边关系                    │
│                                                    │
│  一次编译、持续累积,知识随时间复利增值              │
├──────────────────────────────────────────────────┤
│  Schema(约束层)                                  │
│                                                    │
│  约束 AI 编辑行为的协议(提示词)                    │
│  · 定义每类页面的字段规范                           │
│  · 定义输出格式                                    │
│  · 定义 AI 操作指令集                              │
│  (本知识库的 AGENTS.md 即为 Schema)                │
└──────────────────────────────────────────────────┘
```

### 用做菜类比

| 层 | 类比 | 说明 |
|---|---|---|
| Raw Sources | 食材原产地 | 土豆在地里、鱼在海里,散落各处,没加工 |
| Wiki | 中央厨房 | 食材已洗净、切好、按菜谱分类放好,随用随取 |
| Schema | 菜谱标准 | 规定每道菜用什么食材、什么切法、什么火候 |

## 与传统 RAG 的核心差异

| 维度 | 传统 RAG | LLM Wiki |
|------|----------|----------|
| 处理时机 | 查询时实时检索 | 文档写入时预编译 |
| 解决的核心问题 | 知识定位与精准召回 | 知识质量本身 |
| 原始材料状态 | 不改变,原样存储 | 被 LLM 主动加工、消歧、结构化 |
| 矛盾信息处理 | 无,两份矛盾文档都会被召回 | 以权威来源仲裁,形成唯一真相 |
| 知识积累效应 | 无状态,不随时间增值 | 知识复利,越积累越精准 |
| 运维成本 | 低(存了就行) | 初始高(需要编译),长期趋近于零 |

## 核心价值:知识复利

传统 RAG 每次查询都要重新推导,用完就忘。LLM Wiki 是一次编译、持续累积:

```
传统 RAG:                          LLM Wiki:
  查询1 → 检索 → 生成 → 忘记          编译1 → 页面A
  查询2 → 检索 → 生成 → 忘记          编译2 → 页面B(+ 更新A的交叉引用)
  查询3 → 检索 → 生成 → 忘记          编译3 → 页面C(+ 更新A、B的交叉引用)
  ...                                ...
  每次都是一次性的                    知识网络越来越密,复利增值
```

人工维护 Wiki 难以坚持,因为"记账式"整理(更新关联、排查矛盾)消耗极大。LLM 不会厌倦,可以同时批量更新多个页面,把知识运维的边际成本压到接近零。

## 不是替代,而是组合

实践中 LLM Wiki 并不替代 RAG,而是充当**知识编译层**。最佳组合:

```
原始文档 ──► LLM 编译 ──► Wiki 页面 ──► RAG 检索 ──► 问答
                              │
                    └─ 高质量语料(干净、结构化、无矛盾)
```

LLM Wiki 提供高质量语料,RAG 提供精准检索。先编译再检索,检索噪声大幅降低。

## 本知识库如何落地

本知识库是 Karpathy 三层理论的落地实现:

| Karpathy 三层 | 本知识库 | 说明 |
|---|---|---|
| Raw Sources | 对话记录、外部文档、实践笔记 | 不入库,经编译后进入 Wiki |
| Wiki | `00-AgentBase/` | 知识库本体,人类看懂 + Agent 当 wiki 读 |
| Schema | `00-AgentBase/AGENTS.md` | 入库规范:怎么编译、写入、更新知识库 |

知识库(`00-AgentBase/`)是整个系统的**大脑**:根目录 `AGENTS.md` 基于大脑的知识去搭建、部署、编排 Agent 环境,`01-06/` 是已生成好的工具箱(拿来即用)。

编译和入库流程详见 [知识库 Schema](../AGENTS.md)。部署、编排与管理见 [根目录 AGENTS.md](../../AGENTS.md)。
