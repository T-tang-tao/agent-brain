# Agent 知识库设计

> 基础概念解释见 [00-human/02-核心术语速览-人类版](../00-human/02-核心术语速览-人类版.md)。
>
> 本文件聚焦本知识库的设计原则和分层结构。

## 一句话解释

Agent 知识库是给人和 Agent 共同读取的知识系统。它不只是向量数据库，而是由 wiki、文档、RAG、Memory、Skill、Prompt、项目资料和实时工具共同组成。

## 什么是 Agent 知识库

Agent 知识库用于回答：

```text
Agent 应该知道什么？
Agent 应该在哪里查？
哪些知识是长期的？
哪些状态必须实时查询？
哪些内容是给人读的？
哪些内容是给 Agent 执行的？
```

本目录 `agent/` 就是一个 Agent 知识库的雏形。

## Wiki、RAG、Memory、Skill 的区别

| 概念 | 作用 |
|---|---|
| Wiki | 人和 Agent 都能读的结构化知识 |
| RAG | 从大量资料中检索当前相关内容 |
| Memory | 跨会话保存长期经验 |
| Skill | 某类任务的操作流程和最佳实践 |
| Prompt | 当前任务或角色的行为指令 |

这些不是同一个东西。

## 人能读和 Agent 能读的知识库有什么不同

人类读者需要：

- 解释；
- 例子；
- 学习顺序；
- 概念对比；
- 清晰标题。

Agent 需要：

- 稳定路径；
- 明确小节；
- 可抽取表格；
- 边界说明；
- 相关文件链接；
- 可执行模板和规则。

一个好的 Agent 知识库应该同时满足两者。

## 不同场景的知识库为什么不同

不同 Agent 任务需要不同知识结构。

不要幻想一个统一向量库解决所有问题。

## Coding Agent 知识库

Coding Agent 需要：

- repo 文件；
- README；
- 架构文档；
- 测试说明；
- issue / PR 背景；
- 代码搜索；
- symbol / dependency 信息；
- 项目约束和历史坑。

代码知识库不能只靠向量检索，因为代码需要精确路径、调用关系和当前文件状态。

更适合：

```text
实时文件读取 + grep/search + 项目 wiki + 测试结果 + 架构笔记
```

## 交易/量化 Agent 知识库

交易/量化场景需要特别注意时间和数据边界。

需要区分：

| 类型 | 建议存放方式 |
|---|---|
| 行情数据 | 时序数据库、Parquet、行情 API |
| 策略说明 | wiki / markdown |
| 因子定义 | feature store / 文档 |
| 回测结果 | 实验记录 / 报告 |
| 风控规则 | 可执行规则和配置 |
| 实时价格 | 工具/API 查询，不写入静态知识库 |
| 账户状态 | 工具/API 查询，不写入静态知识库 |

关键原则：

```text
实时状态不应该写成静态知识。
```

例如价格、账户余额、订单状态应该通过 tool 或 MCP 实时查询。

## 客服/文档问答知识库

客服或文档问答需要：

- 权威来源；
- 版本控制；
- 引用；
- 权限过滤；
- 避免编造；
- 更新机制。

适合：

```text
产品文档 + FAQ + 政策版本 + RAG + 引用来源
```

## 个人助手知识库

个人助手需要：

- 用户偏好；
- 长期习惯；
- 隐私边界；
- 可删除；
- 可纠正。

适合写入 Memory 的内容应该经过筛选，不应把所有对话都记住。

## 本目录的知识库分层

当前 `agent/` 采用两层结构：

| 层级 | 目录 | 说明 |
|---|---|---|
| 知识区 | `00-AgentBase/` | 解释概念和学习基础 |
| 资产区 | `01-*` 之后 | 存放可直接使用的 skill、prompt、MCP、边界规则、迁移 playbook |

## 知识库设计原则

1. **知识和资产分离**：概念解释放 `00-AgentBase/`，可执行资产放 `01-*`。
2. **实时状态用工具查**：不要把价格、订单、账户、最新 API 状态写死。
3. **长期记忆要少而准**：Memory 只保存未来有用的长期信息。
4. **文件要小而清晰**：每篇解决一个问题。
5. **链接要明确**：使用 Obsidian 双链连接相关概念。
6. **边界要写清楚**：Agent 需要知道什么不能做。
7. **模板和规则要可复用**：资产区内容应能直接复制或迁移。

## 常见误解

### 误解 1：知识库就是向量数据库

不是。向量数据库只是检索实现之一。

### 误解 2：所有内容都应该被 Agent 记住

不是。临时信息、敏感信息、可从代码中重新获得的信息，不应随便写入长期记忆。

### 误解 3：文档越多越好

不是。过期、重复、矛盾的文档会让 Agent 更混乱。
